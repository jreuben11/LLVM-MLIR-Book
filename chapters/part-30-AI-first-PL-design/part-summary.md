## Part Overview

Part XXX establishes the theoretical and engineering foundation for a class of programming languages that do not yet fully exist but are urgently needed: languages where the primary author is an AI agent rather than a human. The argument begins with Chapter 203's identification of seven design pressures that invert human-PL conventions — formal semantics over convention, rich type systems as machine-readable specifications, explicit algebraic effects, homoiconicity, canonical single representation, structured typed diagnostics, and module-level edit locality. These are not incremental improvements to existing languages; they are a reorientation of what a programming language is for. The chapter surveys the landscape of existing systems — MoonBit, Pel, Dana as purpose-built AI-first PLs; Dafny, Lean 4, Verus as vericoding languages; Unison, Koka, Idris 2 as structurally AI-affine languages; LMQL, Guidance, Outlines, DSPy as the LLM interaction layer — and maps each to the properties it implements and the gaps it leaves. The five probabilistic programming paradigms from Stan, Pyro, and Gen provide the type-theoretic grounding for stochastic AI behaviour, culminating in a unified bridge between the PPL tradition and the algebraic effects model that subsequent chapters formalise.

Chapter 204 is the specification chapter: it defines the unified AI-first PL across six layers — lexical structure (no indentation sensitivity, mandatory type annotations, canonical formatter), concrete EBNF syntax (effect rows, security labels, tensor dimensions, linearity annotations as distinct grammatical productions), typed AST (content-addressed by SHA-256, with `Quote`/`Unquote` and `Sorry` as first-class nodes), a formal type system with seven type constructors and the judgement form `Γ ; Φ ⊢ e : T ! E`, denotational and operational semantics for `{Stochastic}` and `{Gradient}`, and a seven-phase implementation roadmap. Six soundness corollaries are derived from the type rules: effect safety, stochastic safety, prompt-injection non-leakage (IFT non-leakage), linear conservation, tensor shape safety, and capability safety. The chapter identifies three hardest type-system interactions that must be proved correct before production implementation begins — differentiating a stochastic function, security-labelling proof terms, and linear resources in handler continuations — establishing the Phase 0 gate that prevents repeating the Swift for TensorFlow failure.

Chapter 205 applies the AI-first PL framework to the specific domain of transformer model development. It articulates ten design considerations absent from Python frameworks: named typed tensor dimensions (shape mismatches as type errors, not runtime exceptions), automatic differentiation as a type-level transform with a `{Gradient}` algebraic effect, parallelism strategies as type annotations with compiler-derived collective communication, device and precision as type parameters, compiler-derived kernel fusion, stochasticity as an algebraic effect with swappable handlers, recomputation annotations, first-class einsum syntax, MLIR as the compilation target, and LLM-legible compact kernel DSLs with Speed-of-Light performance feedback. The chapter surveys existing systems — Mojo (most production-ready, but lacks dependent tensor types and first-class AD), Dex (closest to the type-theoretic ideal, research-grade), Futhark (uniqueness types, auto-AD), Exo 2 (schedulable kernel authorship via Cursors), Clef (dimensional type safety through MLIR), Burn (Rust-native, no dependent types) — and the emerging body of LLM-driven kernel generation research (AutoKernel, KernelSmith, CuTeGen, SOL-guided DSLs, ACCLAIM). StableHLO/MLIR is identified as the universal convergence IR; the `tensor_named` dialect for named dimension preservation is the key open implementation task.

Chapter 206 addresses the systems-level consequences of AI-first PL design: multi-agent coordination, SDLC redesign, and security. The multi-agent section identifies five properties absent from Python orchestration frameworks — choreographic protocol compilation (Choral's "write global, compile per-agent" model with EPP), probabilistic session types for LLM non-determinism as a first-class algebraic effect, capability-typed delegation with linear resource budgets for token costs, search strategy as a structural type annotation (EnCompass's `@branchpoint`), and intent-level typing for semantic divergence prevention. The AI-first SDLC section reimagines every human-cognitive artifact: work items as `Spec[T]` with content-addressed identity, commits as type-context deltas `Commit[Γ₁, Γ₂]` with compiler-derived semver labels, deployment manifests derived from tensor shape type annotations, and CI/CD as a formal state machine with proof-discharge merge gates. The security section derives three principles — Pony's reference capabilities for data-race-free concurrency, E's no-ambient-authority model for tool handles, and Jif's information-flow type system making prompt injection a type error — and shows how they compose with the algebraic effects system. The paradigm retrospective reframes every canonical software engineering failure (No Silver Bullet, UML round-trip, formal methods retreat, KISS) as a human-cost problem that AI eliminates, not a theoretical problem.

Chapter 207 synthesises the entire part through three lenses. The reflective code section characterises what fully introspective, homoiconic, self-improving code requires: five properties (static introspection via `MetaM`, dynamic introspection via execution trace types, homoiconicity via `quote`/`unquote`, certified self-modification via `patch : Program → MetaModel → Σ(Program, Proof)`, and RL-based self-improvement with formal oracle), four proof-search paradigms from AlphaProof/DeepSeek-Prover-V2/LeanCopilot/HyperTree, three homoiconicity paradigms from Julia/@generated/Racket/Pharo, and a formal type-theoretic formulation with `describe : Program → MetaModel[P]` and `mape_k : MetaModel P → until(Checkpoint P) → Program`. The open problems section identifies seven cross-cutting gaps ordered by priority: G3 (testing layer as the engineering bridge to near-term adoption), G2 (typed holes for incremental verification), G1 (typed `LLMCall` primitive unifying stochastic, linear-budget, and IFT concerns), G5 (type inference strategy), G4 (multimodal I/O types), G6 (API boundary contracts), and G7 (RLHF/DPO formalisation). The build roadmap provides six concrete phases from a reference interpreter through a reflective substrate, with team sizes, durations, success criteria, and risk assessments.

## Key Concepts Introduced

- **AI-first design pressures**: Seven properties that invert human-PL conventions — formal semantics, rich type systems as specs, explicit algebraic effects, homoiconicity, canonical representation, structured diagnostics, edit locality — establishing what any AI-first PL must provide.
- **Vericoding**: The generation of formally verified code from formal specifications (contrasted with vibecoding), with Dafny achieving 82% LLM success rates on formal specs, Verus 44%, and Lean 4 27% but rapidly rising.
- **Row-polymorphic algebraic effects**: Koka's type system, serving as the reference implementation for all effect-row formulas in the part — `{Stochastic}`, `{Gradient}`, `{RequiresHumanApproval}`, `{HighIntegrity}` — composing additively so compound effect rows like `<Stochastic, Gradient, IO>` are first-class types.
- **Probabilistic PPL bridge**: Five paradigms from Stan, Pyro/NumPyro, and Gen grounding AI probabilistic behaviour in the type system — `Dist[A]` as a type constructor, inference as an algebraic effect handler (swappable by changing the handler, not the model), execution traces as first-class typed values, compiler-derived gradients, and convergence bounds as refinement types.
- **Named typed tensor dimensions**: The type `Tensor[Batch, SeqLen, DModel, Float32]` where dimension names are type-level variables, shape constraints are discharged by Z3 linear arithmetic, and shape mismatches are compile-time errors rather than runtime exceptions.
- **`{Gradient}` as algebraic effect**: Automatic differentiation formalised as a type-level transform — `grad : (A →<Gradient> B) → A → (B → A)` — where functions without `{Gradient}` in their effect row are statically non-differentiable and composing them with differentiable functions is a type error.
- **Content-addressed AST identity**: The Unison model applied to the unified PL — program identity is the SHA-256 hash of the normalised, alpha-reduced AST, making rename and reformat conflict-free, enabling incremental builds from structural content rather than file timestamps.
- **Choreographic programming with EPP**: Choral/HasChor's "write global, compile per-agent" model — a coordination protocol written once from a global perspective, with per-participant implementations derived via endpoint projection — making protocol violations compile-time errors.
- **Linear resource types for token budgets**: Idris 2's Quantitative Type Theory (`T^1`) applied to agent token budgets and GPU tensor ownership, preventing duplication of resources that should be consumed exactly once.
- **Information-flow labels as types**: Jif's Decentralised Label Model where `String{Untrusted} → String{Trusted}` implicit flows are type errors, making prompt injection structurally impossible in well-typed programs without explicit sanitisation.
- **Object capability model / no ambient authority**: E language's principle that authority to perform an action is a first-class typed value held explicitly in the function's effect row — MCP tool handles as capability references, not ambient globals, enforced by the `Unquote` type rule.
- **`patch` with proof**: The type signature `patch : Π(P: Program, M: MetaModel P) → Σ(Q: Program, Q satisfies spec P)` capturing certified self-modification — the patched program carries a proof of invariant preservation, with no production implementation yet.
- **`mape_k` typed self-adaptation loop**: A formally typed Monitor-Analyze-Plan-Execute-K loop expressed as `mape_k : MetaModel P → until(Checkpoint P) → Program`, connecting runtime trace observations to formally verified patches via the dependent type system.
- **Typed `LLMCall` primitive (G1)**: The unified first-class type `LLMCall[Model, In: Schema, Out: Schema, Budget: TokenCount] : {Stochastic, Billed[Budget], IO} Out` connecting FSM-constrained output schemas (Outlines), swappable stochastic handlers, linear token budgets, and IFT security labels in a single type.
- **Proof-gated CI/CD**: A formal state machine for the software development lifecycle where the merge gate is formal verification discharge (`Spec[T]` postconditions satisfied), not human approval — human review reserved for policy decisions requiring `{RequiresHumanApproval}` capability grants.
