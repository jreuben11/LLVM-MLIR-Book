# Chapter 178 — The Rust Compiler Ecosystem

*Part XXVI — Ecosystem and Frontiers*

The Rust community has built a remarkably productive layer of compiler infrastructure on top of LLVM and MLIR's C APIs. Where C++ consumers of LLVM must navigate the full library surface directly, Rust crates provide both raw unsafe bindings and carefully lifetime-typed safe wrappers. The result is a coherent ecosystem spanning LLVM IR generation, MLIR construction and transformation, object-file and DWARF manipulation, type-inference utilities, and GPU kernel authoring — all expressible in a single language that the compiler reasons about statically. The crates surveyed in this chapter are not toy bindings: `gimli` and `object` are used by the Rust compiler itself; `ena` was extracted from rustc's type inference engine; `inkwell` underpins production JIT compilers; and CubeCL generates the GPU kernels that power Burn, a full-featured ML training framework. Understanding this ecosystem is essential for any engineer building compiler infrastructure who is choosing between C++, Rust, or a polyglot approach. Each section covers the crate's design philosophy, API patterns, and the tradeoffs it makes against the C++ counterpart.

## 178.1 LLVM Bindings: llvm-sys and inkwell

### llvm-sys: Raw Bindings to the LLVM C API

[llvm-sys](https://crates.io/crates/llvm-sys) provides mechanical, `bindgen`-generated bindings to LLVM's stable C API — the same API that the LLVM project commits to keeping compatible across minor releases. Every function in `llvm-c/*.h` is exposed as an `unsafe extern "C"` declaration, re-exported through module paths that mirror the header organization: `llvm_sys::core`, `llvm_sys::analysis`, `llvm_sys::execution_engine`, `llvm_sys::target_machine`, and so on. The generated source is checked into the crate so that consumers do not need `clang`/`bindgen` at build time.

**Dependency and version selection.** The crate encodes the LLVM major version in both its crate version number and a feature flag. Edition 2024 `Cargo.toml`:

```toml
[package]
name    = "my-jit"
version = "0.1.0"
edition = "2024"

[dependencies]
llvm-sys = { version = "220", features = ["llvm22-0"] }
```

The `build.rs` generated by `llvm-sys` invokes `llvm-config --libs --ldflags --cflags` to locate the installed LLVM and emit the appropriate `-L` and `-l` directives to Cargo. If `LLVM_SYS_220_PREFIX` is set in the environment it overrides the system installation, enabling CI builds against a pinned LLVM tree or a custom build directory. The feature flag and crate version must agree with the installed LLVM; the build script emits a compile error if the `llvm-config --version` output does not match the requested version, preventing silent ABI mismatch.

When multiple LLVM versions are installed, `llvm-config` resolution follows `PATH` order, or you can set `LLVM_SYS_220_PREFIX=/usr/lib/llvm-22` explicitly. If your project links both `llvm-sys` and a crate that also links LLVM (e.g., an MLIR binding crate), ensure they agree on the version to avoid duplicate symbol linker errors.

**Edition 2024 `unsafe extern` blocks.** Edition 2024 requires that `extern "C"` blocks containing FFI declarations be prefixed with `unsafe`. This is a soundness tightening: unsafety is now visible at the declaration site rather than being implied only at call sites. When writing thin wrappers over `llvm-sys` that call into additional C API functions not yet covered by the crate's bindings, the Edition 2024 form is:

```rust
unsafe extern "C" {
    fn LLVMVerifyFunction(Fn: LLVMValueRef, Action: LLVMVerifierFailureAction) -> LLVMBool;
    fn LLVMViewFunctionCFGOnly(Fn: LLVMValueRef);
}
```

The `llvm-sys` crate itself is regenerated with Edition 2024 semantics; all its declarations use the new form.

**Hello world: create a module and dump IR.** The program below creates an LLVM module, adds a single `i32 @main()` returning 0, and prints the textual IR to stderr via `LLVMDumpModule`. This demonstrates the full lifecycle of an `llvm-sys` program: allocate context, build IR, dump it, then tear down in reverse order.

```rust
use llvm_sys::core::*;
use llvm_sys::prelude::*;

fn main() {
    unsafe {
        let context = LLVMContextCreate();
        let module  = LLVMModuleCreateWithNameInContext(
            c"hello".as_ptr(),
            context,
        );
        let i32_ty  = LLVMInt32TypeInContext(context);
        let fn_ty   = LLVMFunctionType(i32_ty, std::ptr::null_mut(), 0, 0);
        let func    = LLVMAddFunction(module, c"main".as_ptr(), fn_ty);
        let bb      = LLVMAppendBasicBlockInContext(context, func, c"entry".as_ptr());

        let builder = LLVMCreateBuilderInContext(context);
        LLVMPositionBuilderAtEnd(builder, bb);
        let zero = LLVMConstInt(i32_ty, 0, 0);
        LLVMBuildRet(builder, zero);

        LLVMDumpModule(module);

        LLVMDisposeBuilder(builder);
        LLVMDisposeModule(module);
        LLVMContextDispose(context);
    }
}
```

`c"hello"` is a C-string literal available since Rust 1.77 and idiomatic in Edition 2024; it yields a `&CStr` with no heap allocation, eliminating the `CString::new(...).unwrap().as_ptr()` pattern. The teardown order matters: the builder must be disposed before the module, and the module before the context. Failing to call any `LLVMDispose*` causes a memory leak; calling them out of order causes a use-after-free. `llvm-sys` cannot enforce either constraint — this is the primary motivation for inkwell.

Running this binary produces:

```
; ModuleID = 'hello'
source_filename = "hello"

define i32 @main() {
entry:
  ret i32 0
}
```

**Extending with the analysis API.** After building IR, you typically want to verify it and optionally print diagnostic text. The C API provides `LLVMVerifyModule` from `llvm_sys::analysis`:

```rust
use llvm_sys::analysis::{LLVMVerifyModule, LLVMVerifierFailureAction};
use std::ptr;

unsafe {
    let mut err: *mut i8 = ptr::null_mut();
    let failed = LLVMVerifyModule(
        module,
        LLVMVerifierFailureAction::LLVMPrintMessageAction,
        &mut err,
    );
    if failed != 0 {
        eprintln!("Verification failed");
    }
    if !err.is_null() {
        LLVMDisposeMessage(err);
    }
}
```

**Building a minimal RAII layer.** Production use of `llvm-sys` requires wrapping the raw handles in types that call their `LLVMDispose*` counterparts on `Drop`. A pattern that works well is a newtype that owns the handle and exposes a `borrow` method returning a reference type with a shorter lifetime:

```rust
pub struct OwnedModule(LLVMModuleRef);

impl OwnedModule {
    pub fn new(name: &CStr, context: LLVMContextRef) -> Self {
        Self(unsafe { LLVMModuleCreateWithNameInContext(name.as_ptr(), context) })
    }
    pub fn as_ref(&self) -> LLVMModuleRef { self.0 }
}

impl Drop for OwnedModule {
    fn drop(&mut self) { unsafe { LLVMDisposeModule(self.0); } }
}
```

The key discipline: `OwnedModule` must not outlive the `LLVMContextRef` it was created with. If your RAII layer groups context + module + builder into a single struct, the compiler enforces drop order automatically (fields are dropped in declaration order). For more complex dependency graphs — e.g., multiple modules sharing one context, or a module handed to an execution engine that then owns it — you need explicit lifetime tracking or reference counting. inkwell handles this through the `'ctx` phantom lifetime; rolling a custom solution means you are reimplementing what inkwell already provides.

**Design tradeoff.** `llvm-sys` is appropriate when you need fine-grained control, need to reach LLVM API surface that higher-level wrappers have not yet exposed, or are building a wrapper crate yourself. Every value is a raw pointer (`LLVMValueRef`, `LLVMTypeRef`, `LLVMModuleRef`); use-after-free, double-free, and mismatched-context errors are entirely your responsibility. For anything larger than a quick experiment, a thin RAII layer or the full inkwell wrapper is the correct choice.

### inkwell: Safe, Idiomatic LLVM Bindings

[inkwell](https://github.com/TheDan64/inkwell) wraps `llvm-sys` in safe Rust types. The central insight is that LLVM's C API is largely memory-safe if one invariant is enforced: no IR value may outlive its owning `LLVMContextRef`. inkwell enforces this at the type level by threading a `'ctx` phantom lifetime through every IR type, allowing the Rust borrow checker to reject violations at compile time with zero runtime cost.

```toml
[package]
name    = "inkwell-example"
version = "0.1.0"
edition = "2024"

[dependencies]
inkwell = { version = "0.5", features = ["llvm22-0"] }
```

**Core types and the `'ctx` lifetime.** The type hierarchy mirrors LLVM's own object model:

| Type | Lifetime parameter | Description |
|------|--------------------|-------------|
| `Context` | owned (no lifetime) | Root of the ownership graph; its `Drop` impl calls `LLVMContextDispose` |
| `Module<'ctx>` | `'ctx` | Borrowed view of a module; `Context::create_module` returns this |
| `Builder<'ctx>` | `'ctx` | Instruction builder; repositioned via `position_at_end` |
| `FunctionValue<'ctx>` | `'ctx` | Handle to a function definition |
| `BasicBlock<'ctx>` | `'ctx` | A basic block; block arguments are `BasicValueEnum<'ctx>` |
| `IntValue<'ctx>` | `'ctx` | Typed integer SSA value |
| `FloatValue<'ctx>` | `'ctx` | Typed float SSA value |
| `PointerValue<'ctx>` | `'ctx` | Opaque pointer value (post-opaque-pointers) |

The `'ctx` parameter is not a reference to the `Context` struct — it is a phantom lifetime marker. The compiler uses it solely to ensure that `FunctionValue<'ctx>` and friends cannot escape the scope in which the owning `Context` is live. No runtime pointer is stored; no fat pointer or vtable is involved. The cost at runtime is zero.

**JIT-compile and execute an `add` function.** The example below constructs `fn add(a: i64, b: i64) -> i64 { a + b }`, compiles it with the LLVM MCJIT execution engine, and calls it through a typed function pointer retrieved by name.

```rust
use inkwell::context::Context;
use inkwell::execution_engine::JitFunction;
use inkwell::OptimizationLevel;

type AddFunc = unsafe extern "C" fn(i64, i64) -> i64;

fn main() {
    let context  = Context::create();
    let module   = context.create_module("add_mod");
    let builder  = context.create_builder();

    let i64_ty   = context.i64_type();
    let fn_ty    = i64_ty.fn_type(&[i64_ty.into(), i64_ty.into()], false);
    let function = module.add_function("add", fn_ty, None);
    let bb       = context.append_basic_block(function, "entry");

    builder.position_at_end(bb);
    let a   = function.get_nth_param(0).unwrap().into_int_value();
    let b   = function.get_nth_param(1).unwrap().into_int_value();
    let sum = builder.build_int_add(a, b, "sum").unwrap();
    builder.build_return(Some(&sum)).unwrap();

    let engine = module
        .create_jit_execution_engine(OptimizationLevel::None)
        .unwrap();

    let add_fn: JitFunction<AddFunc> = unsafe {
        engine.get_function("add").unwrap()
    };

    let result = unsafe { add_fn.call(3, 4) };
    assert_eq!(result, 7);
    println!("add(3, 4) = {result}");
}
```

Notice how `FunctionValue<'_>`, `IntValue<'_>`, and `BasicBlock<'_>` all carry the context's lifetime implicitly — the compiler infers them all as `'ctx` and rejects any attempt to store them past the owning `Context`. `JitFunction<'ctx, F>` ties the function pointer to the execution engine's lifetime, preventing the pointer from outliving the JIT module. This is the key ergonomic advantage over `llvm-sys`: the borrow checker enforces the same invariants that you would otherwise need to track mentally.

`builder.build_int_add` returns `Result<IntValue<'ctx>, BuilderError>`; the `.unwrap()` calls are acceptable in examples but production code should propagate via `?`. The `BuilderError` variants cover the cases where the builder is not positioned at a valid insertion point.

**Optimization passes via the New Pass Manager.** inkwell exposes the NPM through `Module::run_passes`, accepting the same pipeline string syntax as `opt -passes=`:

```rust
use inkwell::passes::PassBuilderOptions;
use inkwell::targets::{InitializationConfig, Target, TargetMachine, RelocMode, CodeModel, FileType};
use inkwell::OptimizationLevel;

Target::initialize_native(&InitializationConfig::default()).unwrap();
let target_triple  = TargetMachine::get_default_triple();
let target         = Target::from_triple(&target_triple).unwrap();
let target_machine = target
    .create_target_machine(
        &target_triple,
        "generic",
        "",
        OptimizationLevel::Default,
        RelocMode::Default,
        CodeModel::Default,
    )
    .unwrap();

let opts = PassBuilderOptions::create();
module.run_passes(
    "mem2reg,instcombine,reassociate,gvn,simplifycfg",
    &target_machine,
    opts,
).unwrap();
```

The pipeline string is passed directly to `llvm::PassBuilder::parsePassPipeline`; any pipeline that `opt` accepts works here, including nested pipelines like `default<O2>` or `function(mem2reg,instcombine)`.

**Global variables and module-level constants.** inkwell exposes global variables through `module.add_global(ty, address_space, name)`, returning a `GlobalValue<'ctx>`. Setting the initializer and linkage follows the same pattern as the C++ API:

```rust
use inkwell::module::Linkage;
use inkwell::values::GlobalValue;

let i32_ty     = context.i32_type();
let global: GlobalValue<'_> = module.add_global(i32_ty, None, "ANSWER");
global.set_initializer(&i32_ty.const_int(42, false));
global.set_linkage(Linkage::External);
global.set_constant(true);
```

`address_space` is `None` for the default address space (0) or `Some(AddressSpace::from(n))` for target-specific address spaces (e.g., CUDA shared memory at address space 3). Thread-local globals use `global.set_thread_local_mode(ThreadLocalMode::GeneralDynamic)`.

**Struct types and aggregate construction.** Named struct types are registered with the context and used across functions; anonymous struct types are created inline:

```rust
let i32_ty  = context.i32_type();
let ptr_ty  = context.ptr_type(AddressSpace::default());
let pair_ty = context.struct_type(&[i32_ty.into(), ptr_ty.into()], false);

let named = context.opaque_struct_type("MyStruct");
named.set_body(&[i32_ty.into(), i32_ty.into()], false);
```

Constant struct values are constructed via `context.const_struct(&[val1, val2], false)` or `pair_ty.const_named_struct(&[val1, val2])`.

**Emitting object code.** Once a module is optimized, emitting native object code for on-disk linking uses the same `TargetMachine`:

```rust
target_machine
    .write_to_file(&module, FileType::Object, std::path::Path::new("output.o"))
    .unwrap();
```

This produces a relocatable ELF (or Mach-O on macOS, COFF on Windows) that can be passed directly to a system linker. For in-memory linking, `write_to_memory_buffer` returns an `inkwell::memory_buffer::MemoryBuffer` that you can hand to JITLink or an embedded linker like LLD.

### iron-kaleidoscope: A Tutorial Compiler in Rust

[iron-kaleidoscope](https://github.com/jauhien/iron-kaleidoscope) is a full implementation of the LLVM Kaleidoscope tutorial ported from C++ to Rust using inkwell. It covers the complete pipeline: lexer, parser, AST construction, code generation to inkwell IR, MCJIT-based execution, and optimization passes applied per-function. The repository is organized chapter by chapter, matching the official LLVM Kaleidoscope tutorial structure, which makes it easy to cross-reference when following the tutorial in Rust.

The key design decisions iron-kaleidoscope demonstrates are worth studying:

- **Codegen context encapsulation.** A `Compiler` struct owns the `Context`, `Module`, `Builder`, and an `FxHashMap<String, PointerValue<'_>>` for the named value environment. This is the idiomatic way to pass inkwell types through a recursive descent codegen without fighting the borrow checker.
- **Per-function optimization.** The tutorial applies `mem2reg` and `instcombine` immediately after each function is code-generated, before JIT compilation. This matches how Clang's `-O1` pipeline works for straightforward scalar code.
- **JIT and re-evaluation.** The REPL pattern — parse, codegen, JIT, execute — is cleanly separated from the module/engine lifetime management.

iron-kaleidoscope targets an older inkwell API (pre-0.4) and does not compile against inkwell 0.5 or LLVM 22 bindings without porting. The required changes are:

- Replace `inkwell::passes::PassManager::create_for_function` (removed in the legacy pass manager deprecation) with `module.run_passes(...)` using the NPM.
- Update `ExecutionEngine::get_function` to handle the `Result<JitFunction<F>, FunctionLookupError>` return type.
- Propagate `BuilderError` from `build_*` methods rather than unwrapping silently.
- Remove uses of the removed `add_verifier_pass` — verification is now done with `module.verify()`.

Treating iron-kaleidoscope as a reference architecture and porting it forward is a worthwhile learning exercise. The structural lessons about separating lexer, parser, AST, codegen context, and JIT lifecycle translate directly to any LLVM-based compiler project.

## 178.2 MLIR Bindings: melior and pliron

### melior: MLIR via the C API

[melior](https://github.com/raviqqe/melior) provides Rust bindings to the MLIR C API (`mlir-c/IR.h`, `mlir-c/Pass.h`, `mlir-c/BuiltinTypes.h`, etc.). The design mirrors inkwell's approach: every MLIR object type is parameterized with a lifetime tied to the `Context`, enforcing the invariant that no operation, block, or region outlives the context that owns them. The `ods-dialects` feature (off by default) generates typed builder functions for ODS-defined operations in the standard dialect set.

```toml
[package]
name    = "melior-example"
version = "0.1.0"
edition = "2024"

[dependencies]
melior = { version = "0.20", features = ["ods-dialects"] }
```

**Key types and the ownership model.** MLIR's C API uses a different ownership model from LLVM's C API. Most `MlirOperation` handles are *not* owned by the caller; they are owned by whatever block they are attached to, which is owned by a region, which is owned by an operation. melior reflects this with two distinct handle types per object:

| melior owned type | melior borrowed type | MLIR concept |
|-------------------|---------------------|-------------|
| `OwnedOperation<'c>` | `OperationRef<'c, '_>` | `MlirOperation` |
| `OwnedModule<'c>` | `ModuleRef<'c, '_>` | top-level module |
| `OwnedBlock<'c>` | `BlockRef<'c, '_>` | `MlirBlock` |
| `OwnedRegion<'c>` | `RegionRef<'c, '_>` | `MlirRegion` |
| `Value<'c, '_>` | (same — always borrowed) | `MlirValue` |
| `Type<'c>` | (same) | `MlirType` |
| `Attribute<'c>` | (same) | `MlirAttribute` |
| `Location<'c>` | (same) | `MlirLocation` |

When you append an `OwnedOperation` to a block, ownership transfers to the block; thereafter you hold only a `OperationRef`. This two-type pattern prevents double-free without runtime reference counting.

**Building and transforming MLIR.** The following constructs a `func.func` containing an `arith.addi`, runs the canonicalizer pass, and prints the result. The `ods-dialects` feature provides the `func::func`, `func::r#return`, and `arith::addi` builder functions, which are far more ergonomic than raw `OperationBuilder` string-keyed construction.

```rust
use melior::{
    Context,
    dialect::{arith, func, DialectRegistry},
    ir::{
        attribute::{StringAttribute, TypeAttribute},
        r#type::FunctionType,
        Block, Module, Region, Location, Type,
    },
    pass::{self, PassManager},
    utility::register_all_dialects,
};

fn main() {
    let registry = DialectRegistry::new();
    register_all_dialects(&registry);

    let context = Context::new();
    context.append_dialect_registry(&registry);
    context.load_all_available_dialects();

    let location = Location::unknown(&context);
    let module   = Module::new(location);

    let i32_ty   = Type::parse(&context, "i32").unwrap();
    let fn_type  = FunctionType::new(&context, &[i32_ty, i32_ty], &[i32_ty]);

    let func_op = func::func(
        &context,
        StringAttribute::new(&context, "add"),
        TypeAttribute::new(fn_type.into()),
        {
            let block = Block::new(&[(i32_ty, location), (i32_ty, location)]);
            let a = block.argument(0).unwrap().into();
            let b = block.argument(1).unwrap().into();
            let sum = block.append_operation(arith::addi(a, b, location));
            let result = sum.result(0).unwrap().into();
            block.append_operation(func::r#return(&context, &[result], location));
            let region = Region::new();
            region.append_block(block);
            region
        },
        &[],
        location,
    );
    module.body().append_operation(func_op);

    let pm = PassManager::new(&context);
    pm.add_pass(pass::transform::create_canonicalizer());
    pm.run(&mut module.as_operation()).unwrap();

    println!("{}", module.as_operation());
}
```

After canonicalization the module prints:

```mlir
module {
  func.func @add(%arg0: i32, %arg1: i32) -> i32 {
    %0 = arith.addi %arg0, %arg1 : i32
    return %0 : i32
  }
}
```

**Pass management.** `PassManager::new(&context)` wraps `mlirPassManagerCreate`. Passes are added either by handle — the `melior::pass::*` module re-exports factory functions for all in-tree MLIR passes — or by pipeline string via `pm.add_pipeline("canonicalize,cse")`. Nested pipeline managers scoped to specific op types are created with `pm.nested_under("func.func")`, which registers passes that only run on `func.func` operations inside the module. After `pm.run`, the module is mutated in place; melior uses `&mut module.as_operation()` rather than returning a new module.

**Dialect registration for custom C++ dialects.** Projects that define custom MLIR dialects in C++ and expose them via the MLIR C API extension mechanism (`mlirRegisterAllCustomDialects`) can integrate with melior through `DialectHandle::register_dialect(&registry)`. This is the correct integration path for hybrid Rust/C++ compilers where the core dialect is written in C++ tablegen/ODS but the driver, optimization pipeline, and tooling are written in Rust.

**Type parsing as an escape hatch.** `Type::parse(&context, "!my_dialect.my_type<i32>")` bypasses the typed API and falls back to MLIR's textual type parser. This is useful for custom types not yet wrapped in melior's ODS layer and for round-tripping types through textual MLIR.

**Raw `OperationBuilder` for non-ODS dialects.** When `ods-dialects` is not enabled or you are targeting a custom dialect without generated builders, `OperationBuilder` provides a string-keyed construction API:

```rust
use melior::ir::{
    operation::OperationBuilder,
    Attribute, Block, Location, Type, Value,
};
use melior::Context;

fn build_raw_addi<'c>(
    ctx:  &'c Context,
    lhs:  Value<'c, '_>,
    rhs:  Value<'c, '_>,
    loc:  Location<'c>,
) -> melior::ir::Operation<'c> {
    let i32_ty = Type::parse(ctx, "i32").unwrap();
    OperationBuilder::new("arith.addi", loc)
        .add_operands(&[lhs, rhs])
        .add_results(&[i32_ty])
        .build()
        .unwrap()
}
```

`OperationBuilder::new` takes the op mnemonic as a string; `.add_attributes(&[("value", attr)])` attaches named attributes; `.add_regions(vec![region])` attaches nested regions. This is the fallback path for custom C++ dialects that expose operations through the MLIR C API but have no melior ODS layer.

**Attribute types.** melior's `melior::ir::attribute` module wraps the full MLIR attribute type hierarchy:

| melior type | MLIR attribute |
|-------------|----------------|
| `IntegerAttribute<'c>` | `IntegerAttr` — `i64` value + `IntegerType` |
| `FloatAttribute<'c>` | `FloatAttr` — `f64` value + float type |
| `StringAttribute<'c>` | `StringAttr` — byte string |
| `TypeAttribute<'c>` | `TypeAttr` — type-as-attribute |
| `ArrayAttribute<'c>` | `ArrayAttr` — heterogeneous attribute list |
| `DenseElementsAttribute<'c>` | `DenseElementsAttr` — packed tensor constants |
| `FlatSymbolRefAttribute<'c>` | `FlatSymbolRefAttr` — cross-op symbol reference |

`DenseElementsAttribute` is particularly important for ML applications: it encodes constant weight tensors packed into a byte buffer without the overhead of an `ArrayAttr` list. Constructing one requires a `ShapedType` (the tensor or vector type) and a byte buffer matching the element type's ABI size.

### pliron: A Native Rust IR Framework

[pliron](https://github.com/vaivaswatha/pliron) takes a fundamentally different approach: rather than binding to MLIR's C API, it reimplements MLIR's structural ideas natively in Rust using trait dispatch. The result is a framework for building compiler IRs without any C++ dependency, trading MLIR wire compatibility for ergonomic Rust-native design and a clean ownership model that fits Rust's borrow checker naturally.

```toml
[package]
name    = "pliron-example"
version = "0.1.0"
edition = "2024"

[dependencies]
pliron = "0.8"
```

**Core abstractions.** pliron's design is trait-centric:

- `Op` trait: implemented by every concrete operation type. Provides `get_operation(&self) -> &Operation`, access to operands, results, regions, and blocks, plus `verify` and `print` hooks called by the framework.
- `Type` trait: implemented by each IR type. Types are interned in the `Context` arena and compared by identity.
- `Attribute` trait: implemented by each compile-time constant. Attributes are also interned.
- `Context`: the arena allocating all IR objects; holds the dialect registry mapping names to `Op`/`Type`/`Attribute` factories.
- `OpObj` / `TypePtr<T>` / `AttrObj`: type-erased handles wrapping a pointer into the arena. Downcasting uses `op.downcast_ref::<ConcreteOp>()` backed by `TypeId` comparison.
- `Dialect`: a collection of `Op`/`Type`/`Attribute` registrations, identified by a namespace string.

The use of `TypeId`-based downcasting rather than virtual dispatch through a vtable means pliron can add new operations without modifying a central enum, exactly like MLIR's open operation set.

**Defining a minimal dialect.** The following defines a dialect with one operation, `my.const_int`, that holds an integer value as an attribute:

```rust
use pliron::{
    context::Context,
    dialect::Dialect,
    dialects::builtin::attributes::IntegerAttr,
    dialects::builtin::types::IntegerType,
    decl_op, impl_verify_succ,
    op::Op,
    result::Result,
};

decl_op!(ConstIntOp, "const.int", MyDialect);

impl ConstIntOp {
    pub fn new(ctx: &Context, value: u64, width: u32) -> Self {
        let i_ty = IntegerType::get(ctx, width);
        let attr = IntegerAttr::new(i_ty.into(), value);
        Self::build(
            ctx,
            vec![],
            vec![i_ty.into()],
            vec![("value".into(), attr.into())],
        )
    }

    pub fn get_value(&self, ctx: &Context) -> u64 {
        self.get_operation()
            .get_attr("value")
            .unwrap()
            .downcast_ref::<IntegerAttr>(ctx)
            .unwrap()
            .val()
    }
}

impl_verify_succ!(ConstIntOp);

pub struct MyDialect;

impl Dialect for MyDialect {
    fn name(&self) -> &str { "my" }
    fn register(&self, ctx: &mut Context) {
        ConstIntOp::register(ctx, ConstIntOp::parser_fn);
    }
}
```

`decl_op!` generates the boilerplate that registers the concrete Rust type with the framework's op registry under the name `"my.const_int"`. The `impl_verify_succ!` macro generates a `verify` implementation that always succeeds — real ops would check type constraints here.

Building and traversing IR follows the same block/region/operation hierarchy as MLIR:

```rust
fn demo(ctx: &mut Context) -> Result<()> {
    MyDialect.register(ctx);
    let op = ConstIntOp::new(ctx, 42, 32);
    op.verify(ctx)?;
    println!("{}", op.get_value(ctx));
    Ok(())
}
```

**IR traversal and rewriting in pliron.** pliron provides a walker API similar to MLIR's `walk` on `Operation *`. The `Op::walk` method visits all operations in the IR tree in pre-order or post-order:

```rust
use pliron::op::Op;
use pliron::context::Context;

fn count_ops(ctx: &Context, root: &dyn Op) -> usize {
    let mut count = 0usize;
    root.get_operation().walk(ctx, &mut |op: &dyn Op| {
        count += 1;
        pliron::walkresult::WalkResult::Advance
    });
    count
}
```

Rewriting is done by building replacement operations and splicing them into the IR using block/region mutation methods: `block.insert_op_before(new_op, anchor)` inserts `new_op` immediately before `anchor`; `block.remove_op(op)` detaches an op without destroying it, allowing it to be re-inserted elsewhere. Unlike MLIR's `PatternRewriter` (which enforces rewrite legality via a `ConversionTarget`), pliron's mutation API is direct — legality checking is the caller's responsibility, enforced by `Op::verify` called explicitly after transformation.

**The difference from MLIR's pattern infrastructure.** MLIR's PDL/PDLL pattern matching compiles patterns to a bytecode interpreter with automatic ordering and conflict resolution. pliron has no equivalent: pattern matching is expressed as Rust code iterating over the operation tree and checking `TypeId`-based downcasts. For small compilers and DSLs, this is simpler and more debuggable than PDL; for large production compilers that want automatic pattern priority resolution, it is a gap.

**Design tradeoffs vs. melior.** pliron does not produce MLIR-compatible textual IR or bytecode; `mlir-opt` cannot consume its output. It is the right choice for new compiler projects that want MLIR-style structural discipline — nested regions, block arguments, typed SSA values, an open operation set — without a C++ build dependency or a requirement that LLVM/MLIR be installed. For projects that need to hand off IR to existing MLIR passes, consume StableHLO, or interoperate with the MLIR Python bindings, melior is the correct choice. pliron and melior can coexist in the same workspace if the project generates IR in pliron, serializes it textually, and re-parses it into melior for transformation — though this round-trip approach is awkward and suggests rethinking the architecture if it becomes a pattern.

### Calyxir: Hardware Compiler IR

[Calyx](https://docs.calyxir.org/) is a compiler IR and framework targeting hardware accelerators — FPGAs and custom ASICs. The Calyx language expresses hardware as a composition of *components* (hardware modules), *cells* (sub-components instantiated inside a component), *groups* (sets of guarded assignments controlled by `go`/`done` signals), and *control* (sequential composition, parallel composition, `if`, `while`, and finite state machine schedules over groups).

```
component add_one(in: 32) -> (out: 32) {
  cells {
    add = std_add(32);
    one = std_const(32, 1);
  }
  wires {
    group do_add {
      add.left  = in;
      add.right = one.out;
      out       = add.out;
      do_add[done] = 1'd1;
    }
  }
  control {
    seq { do_add; }
  }
}
```

The key insight is that Calyx separates the *structural* (wires, cells) from the *behavioral* (control) — precisely the distinction that high-level synthesis (HLS) tools blur. By making the separation explicit and verifiable, Calyx can prove correctness properties about the generated hardware that raw HLS tools cannot.

**Lowering to RTL.** Calyx lowers to synthesizable SystemVerilog (or FIRRTL) via CIRCT's `calyx-to-hw` pass. The `calyx-to-hw` pass is a dialect conversion that rewrites each Calyx component into `hw.module`, each group into combinational logic guarded by FSM state, and the control schedule into an FSM whose transitions correspond to the group completion signals. The resulting `hw.module` ops can then be lowered further through `hw-to-sv` to emit `.sv` files ready for Vivado, Quartus, or Verilator.

The Calyx compiler itself is written in Rust. Its internal IR is a Rust data structure — not melior, not pliron — because it predates both. The compiler pipeline is organized as a sequence of passes (`Pass` trait implementations) over the Calyx IR, each transforming components, cells, or the control tree. Passes include: *static timing inference* (replacing `go`/`done` signals with static delay annotations for groups that have known latency), *resource sharing* (merging cells with non-overlapping active intervals), *well-formedness checking* (ensuring all ports are driven on all paths through the control schedule), and *lowering of parallel control* into explicit FSM states. The final lowering to CIRCT is a translation from the Calyx Rust IR to the MLIR CIRCT `calyx` dialect, at which point CIRCT's existing passes handle the remainder.

**Verification properties.** Calyx's design makes several properties verifiable that raw HLS tools cannot guarantee:

- *Completeness*: every output port is driven to a defined value in every cycle during which the group is active.
- *Non-interference*: no two concurrently active groups drive the same wire (detected by the well-formedness checker).
- *Termination*: the control schedule is a directed acyclic graph of groups (excluding `while`); `while` loops require a user-provided invariant annotation for full verification.

These properties are checked statically before RTL emission, so synthesis failures due to multiple drivers or undriven ports — a common source of silent bugs in HLS-generated Verilog — cannot occur in correct Calyx programs.

**Place in the ML-to-FPGA pipeline.** Dahlia, a high-level HLS language, compiles to Calyx as its target IR. ML models expressed in PyTorch can be lowered through `torch-mlir` → `linalg` → `affine` → Calyx for FPGA deployment. The Calyx runtime (for simulation via Icarus Verilog or Verilator) provides a software model of the generated hardware, enabling rapid iteration without a full synthesis run. This pipeline is used by the [Dahlia/Calyx academic project](https://capra.cs.cornell.edu/calyx/) to generate verified FPGA accelerators from high-level programs. Calyx is not a general-purpose IR but the leading approach for verified hardware generation from MLIR-level computation.

## 178.3 Object-File and Debug-Info Crates

### object: Portable Binary Parsing and Writing

[object](https://docs.rs/object) provides a unified API for reading and writing ELF, Mach-O, PE/COFF, WebAssembly, and XCOFF binary files. Reading is zero-copy: `File::parse` accepts a `&[u8]` (typically a memory-mapped region) and returns parsed structures that hold references into that slice, not copies of strings or section data.

```toml
[package]
name    = "read-elf"
version = "0.1.0"
edition = "2024"

[dependencies]
object  = "0.36"
memmap2 = "0.9"
```

**Reading ELF sections.** Given a path to any ELF binary, the following iterates sections and prints each one's name, virtual address, and size:

```rust
use object::{File, Object, ObjectSection};
use std::{fs, path::Path};

fn print_sections(path: &Path) -> object::Result<()> {
    let data = fs::read(path).map_err(|_| object::Error::InvalidFileType)?;
    let file = File::parse(&*data)?;
    println!("{:<30} {:>18} {:>10} {}", "Section", "VAddr", "Size", "Kind");
    for section in file.sections() {
        println!(
            "{:<30} 0x{:016x} {:>10} {:?}",
            section.name().unwrap_or("<unnamed>"),
            section.address(),
            section.size(),
            section.kind(),
        );
    }
    Ok(())
}

fn main() {
    print_sections(Path::new("/proc/self/exe")).unwrap();
}
```

`File::parse` detects the binary format from magic bytes; the `Object` trait provides `sections()`, `symbols()`, `exports()`, `imports()`, `dynamic_relocations()`, and `has_debug_symbols()` through a single polymorphic interface. `SectionKind` is an enum distinguishing `Text`, `Data`, `ReadOnlyData`, `UninitializedData`, `OtherString`, `Debug`, `Linker`, `Note`, `Metadata`, and `Other`, which is more portable than checking ELF `SHT_*` flags directly.

**Accessing section data.** `section.data()` returns `&[u8]` if the section has in-file content (as opposed to BSS). `section.uncompressed_data()` transparently handles SHF_COMPRESSED ELF sections and `__LLVM,__bitcode` Mach-O segments. `section.data_range(addr, size)` extracts a byte range by virtual address, which is what debuggers use when resolving DWARF location expressions.

**Writing object files.** The write side is imperative: construct a `write::Object`, add sections, attach symbols, and call `write()` to serialize.

```rust
use object::write::{Object, StandardSection, Symbol, SymbolSection};
use object::{Architecture, BinaryFormat, Endianness, SymbolKind, SymbolScope};

fn make_object() -> Vec<u8> {
    let mut obj = Object::new(
        BinaryFormat::Elf,
        Architecture::X86_64,
        Endianness::Little,
    );
    let text   = obj.section_id(StandardSection::Text);
    let code: &[u8] = &[0xb8, 0x00, 0x00, 0x00, 0x00, 0xc3]; // mov eax, 0; ret
    let offset = obj.append_section_data(text, code, 16);
    obj.add_symbol(Symbol {
        name:    b"answer".to_vec(),
        value:   offset,
        size:    code.len() as u64,
        kind:    SymbolKind::Text,
        scope:   SymbolScope::Linkage,
        weak:    false,
        section: SymbolSection::Section(text),
        flags:   object::SymbolFlags::None,
    });
    obj.write().unwrap()
}
```

**Symbol and relocation APIs.** `file.symbols()` returns an iterator of `Symbol` handles. For each symbol you can query `symbol.name()`, `symbol.address()`, `symbol.size()`, `symbol.kind()` (`SymbolKind::Text`, `Data`, `Section`, `File`, `Label`, `Tls`, `Unknown`), `symbol.scope()` (`SymbolScope::Unknown`, `Compilation`, `Linkage`, `Dynamic`), and `symbol.is_undefined()`. Filtering to exported text symbols — useful for building a symbol table for a JIT — looks like:

```rust
use object::{Object, ObjectSymbol, SymbolKind, SymbolScope};

for sym in file.symbols() {
    if sym.kind() == SymbolKind::Text
        && sym.scope() == SymbolScope::Linkage
        && !sym.is_undefined()
    {
        println!("{:016x}  {}", sym.address(), sym.name().unwrap_or("?"));
    }
}
```

`file.dynamic_relocations()` iterates ELF `.rela.dyn` entries; `section.relocations()` iterates per-section relocations, returning `(offset, Relocation)` pairs where `Relocation` carries the kind (`RelocationKind::Absolute`, `Relative`, `Got`, `Plt`, …), target symbol index, and addend. This is the API that `cargo-llvm-cov` uses to rebase coverage counter addresses after ASLR.

`object` is used by rustc's self-contained linker (`rustc_codegen_ssa`'s object file emission), `cargo-binutils` (for `nm`, `objdump`, and `strip` wrappers), `wasmparser` (for Wasm object reading), and `probe-rs` (for ELF loading onto microcontrollers). It is the right dependency for any Rust tool that needs to inspect or produce binary artifacts without shelling out to `readelf` or `llvm-objcopy`.

### gimli: Zero-Copy DWARF 5 Reader and Writer

[gimli](https://docs.rs/gimli) is a lazy, zero-copy DWARF 5 reader and writer designed for tools that need only a subset of a binary's debug information — profilers, symbolizers, debuggers, and coverage tools. "Lazy" means gimli only parses bytes you explicitly access: abbreviation tables are memoized per compilation unit, but individual debugging information entries (DIEs) are not decoded until iterated.

```toml
[package]
name    = "gimli-example"
version = "0.1.0"
edition = "2024"

[dependencies]
gimli  = "0.31"
object = "0.36"
```

**Reading DWARF.** `Dwarf::load` accepts a closure mapping `gimli::SectionId` values to byte slices; the standard pattern pairs it with `object` to pull section data:

```rust
use gimli::{Dwarf, EndianSlice, LittleEndian};
use object::{File, Object, ObjectSection};

fn dump_subprograms(elf_data: &[u8]) -> gimli::Result<()> {
    let object = File::parse(elf_data).unwrap();
    let load = |id: gimli::SectionId| -> gimli::Result<EndianSlice<'_, LittleEndian>> {
        let data = object
            .section_by_name(id.name())
            .and_then(|s| s.data().ok())
            .unwrap_or(&[]);
        Ok(EndianSlice::new(data, LittleEndian))
    };
    let dwarf   = Dwarf::load(load)?;
    let mut units = dwarf.units();
    while let Some(header) = units.next()? {
        let unit = dwarf.unit(header)?;
        let mut entries = unit.entries();
        while let Some((_, entry)) = entries.next_dfs()? {
            if entry.tag() == gimli::DW_TAG_subprogram {
                if let Some(attr) = entry.attr_value(gimli::DW_AT_name)? {
                    let name = dwarf.attr_string(&unit, attr)?;
                    println!("subprogram: {}", name.to_string_lossy());
                }
            }
        }
    }
    Ok(())
}
```

`Dwarf::load` is generic over the reader type; `EndianSlice<'_, E>` is the zero-copy option that borrows the original byte slice. If you need owned data (for example when the ELF file is unmapped after loading), `EndianRcSlice` uses reference-counted storage.

**Reading location information and ranges.** DWARF encodes variable locations in `.debug_loc` and `.debug_rnglists` sections. gimli exposes these through `dwarf.locations(&unit, attr)` → `LocationListIter` and `dwarf.ranges(&unit, attr)` → `RngListIter`. Each entry carries a `(start_addr, end_addr, location_expression)` triple. Evaluating a `LocationExpression` uses `gimli::Expression::evaluation(encoding).evaluate()`, which returns a state machine you drive with register values.

**Writing DWARF.** `gimli::write::Dwarf` is a builder API for producing DWARF sections. The write side is used by rustc's `rustc_codegen_ssa` to emit split DWARF (`.dwo`) files; the produced section bytes are then handed to the `object` crate for embedding in an ELF.

```rust
use gimli::write::{
    Address, AttributeValue, DwarfUnit, EndianVec, LineProgram,
    LineString, Sections, UnitEntryId,
};
use gimli::{Encoding, Format, LineEncoding};

fn write_minimal_dwarf() -> Vec<u8> {
    let encoding = Encoding { format: Format::Dwarf32, version: 5, address_size: 8 };
    let line_encoding = LineEncoding::default();
    let mut dwarf = gimli::write::Dwarf::default();
    let line_program = LineProgram::new(encoding, line_encoding,
        LineString::String(b"src/".to_vec()),
        LineString::String(b"main.rs".to_vec()),
        None,
    );
    let mut unit = DwarfUnit::new(encoding);
    unit.unit.line_program = line_program;
    let root = unit.unit.root();
    let subprogram = unit.unit.add(root, gimli::DW_TAG_subprogram);
    unit.unit.get_mut(subprogram).set(
        gimli::DW_AT_name,
        AttributeValue::String(b"main".to_vec()),
    );
    unit.unit.get_mut(subprogram).set(
        gimli::DW_AT_low_pc,
        AttributeValue::Address(Address::Constant(0x1000)),
    );
    unit.unit.get_mut(subprogram).set(
        gimli::DW_AT_high_pc,
        AttributeValue::Udata(0x80),
    );
    dwarf.units.add(unit);
    let mut sections = Sections::new(EndianVec::new(gimli::LittleEndian));
    dwarf.write(&mut sections).unwrap();
    sections.debug_info.0.take()
}
```

**Performance model.** Abbreviation tables are decoded and cached the first time any entry in a given compilation unit is accessed; subsequent entry traversals in that CU pay only the incremental byte-parsing cost. The lazy model is especially valuable for `perf`-based profilers and flamegraph tools: a typical profiling session symbolizes at most a few hundred unique program counter values, touching perhaps 5–10 compilation units out of the hundreds in a large binary.

### addr2line: Address-to-Symbol Resolution

[addr2line](https://docs.rs/addr2line) layers a high-level address-oriented API on top of `gimli`. It precomputes sorted address range tables at construction time and answers location queries in O(log n) via binary search, making it the right choice when the lookup-to-construction ratio is high (debuggers, profilers, panic handlers).

```toml
[package]
name    = "symbolicate"
version = "0.1.0"
edition = "2024"

[dependencies]
addr2line = "0.24"
object    = "0.36"
```

**Resolving an address.** The following loads the current executable and resolves a given address to its source location and function name chain, including inlined frames:

```rust
use addr2line::Context;
use object::{File, Object};
use std::fs;

fn symbolicate(addr: u64) {
    let data = fs::read("/proc/self/exe").unwrap();
    let file = File::parse(&*data).unwrap();
    let ctx  = Context::new(&file).unwrap();

    match ctx.find_location(addr).unwrap() {
        Some(loc) => println!(
            "0x{addr:016x} → {}:{}:{}",
            loc.file.unwrap_or("<unknown>"),
            loc.line.unwrap_or(0),
            loc.column.unwrap_or(0),
        ),
        None => println!("0x{addr:016x} → <no debug info>"),
    }

    let mut frames = ctx.find_frames(addr).unwrap();
    while let Some(frame) = frames.next().unwrap() {
        if let Some(func) = frame.function {
            println!("  in {}", func.demangle().unwrap_or_default());
        }
    }
}
```

`find_frames` is the inlining-aware variant: for a PC inside a deeply inlined call chain, it returns multiple frames — the innermost physical frame first, then each inliner working outward toward the non-inlined function. The Rust `backtrace` crate uses `addr2line` (via `gimli`) for its symbolization step; this is the mechanism behind the file and line information displayed in Rust panic backtraces and in `RUST_BACKTRACE=1` output.

**Construction cost and caching strategy.** `Context::new` traverses `.debug_line` to build address-range tables and caches abbreviation tables for all CUs. On a 10 MB release binary this takes ~5 ms on a modern CPU. After construction, each `find_location` call typically costs < 10 µs including the binary search and DWARF entry decode.

For single-shot command-line tools the construction cost is acceptable. For long-running services that symbolicate stack traces from distributed crash reports — think an error-aggregation service receiving thousands of stack frames per second — caching `Context` objects is essential. The standard pattern is an `Arc<Context>` stored in an `LruCache<(PathBuf, SystemTime), Arc<addr2line::Context>>`, keyed by binary path and last-modification time. The `SystemTime` check detects deployments where a new binary replaces the old one at the same path. `Arc` allows multiple concurrent symbolication threads to share a single `Context` without locking (all `find_location` / `find_frames` calls take `&self`).

For binaries with split DWARF (`.dwo` files or a `.dwp` package), `addr2line::Context::new` does not automatically load the split units; you must use `addr2line::ObjectContext::new_with_sup` and provide the supplementary DWARF object separately. This is the configuration used by production systems that strip main binaries for deployment while retaining `.dwp` files on a debug symbol server.

### ena: Union-Find for Type Inference

[ena](https://docs.rs/ena) is a Rust port of the union-find (disjoint-set) data structure extracted from rustc's type inference engine. It implements unification with snapshot/rollback for speculative unification, optional value association (mapping a type variable to a concrete type once solved), and the congruence closure extension used by higher-level unification algorithms. The crate is small and focused: its single purpose is providing the data structure that makes Hindley-Milner and Damas-Hindley-Milner inference tractable in a constraint-based solver.

```toml
[package]
name    = "ena-example"
version = "0.1.0"
edition = "2024"

[dependencies]
ena = "0.14"
```

**API overview.** The primary type is `UnificationTable<S>`, parameterized by a strategy `S` implementing `UnificationStore`. The two provided strategies are `InPlace<K>` (mutable, backed by a `Vec`) and `Persistent<K>` (immutable, backed by a persistent vector for functional-style solvers). Keys implement `UnifyKey`, providing a `u32` index and an associated `Value` type:

```rust
use ena::unify::{InPlaceUnificationTable, UnifyKey, UnifyValue};

#[derive(Copy, Clone, Debug, PartialEq, Eq)]
struct TyVid(u32);

impl UnifyKey for TyVid {
    type Value = Option<ConcreteType>;
    fn index(&self) -> u32       { self.0 }
    fn from_index(i: u32) -> Self { TyVid(i) }
    fn tag() -> &'static str      { "TyVid" }
}

#[derive(Clone, Debug, PartialEq)]
enum ConcreteType { Int, Float, Bool }

impl UnifyValue for Option<ConcreteType> {
    type Error = ();
    fn unify_values(a: &Self, b: &Self) -> Result<Self, ()> {
        match (a, b) {
            (None, x) | (x, None) => Ok(x.clone()),
            (Some(a), Some(b)) if a == b => Ok(Some(a.clone())),
            _ => Err(()),
        }
    }
}

fn infer_simple_let() {
    let mut table: InPlaceUnificationTable<TyVid> = InPlaceUnificationTable::new();
    let x = table.new_key(None);
    let y = table.new_key(None);
    table.unify_var_var(x, y).unwrap();
    table.unify_var_value(x, Some(ConcreteType::Int)).unwrap();
    assert_eq!(table.probe_value(y), Some(ConcreteType::Int));
    println!("y inferred as {:?}", table.probe_value(y));
}
```

This models `let x = 1; let y = x;` — both `x` and `y` start as unresolved type variables. When `x` is unified with `Int` (from the integer literal's type), `y` inherits that binding through the union-find root, without any additional code. `probe_value` returns the current value of the root representative, which may be `None` (unresolved) or `Some(T)` (solved).

**Snapshot and rollback.** The killer feature for backtracking solvers is O(1) snapshot/rollback:

```rust
let snap = table.snapshot();
match table.unify_var_value(x, Some(ConcreteType::Float)) {
    Ok(()) => {
        table.commit(snap);
    }
    Err(()) => {
        table.rollback_to(snap);
    }
}
```

`snapshot()` records the current length of the internal undo log; `rollback_to` replays the undo log backwards to the snapshot point. All union and value assignments between snapshot and rollback are undone without any allocation beyond the undo log entries appended during the speculative region. This O(1) rollback is what makes backtracking trait solving tractable: the Chalk trait resolver speculatively applies impl candidates and rolls back on failure, potentially thousands of times per query.

**Congruence closure.** ena's `UnifyValue::unify_values` function is the hook for implementing congruence closure — the rule that if `f(a) = f(b)` and `a = b` then the two applications are equivalent. A type inference engine uses this when unifying generic instantiations: if `Option<T1>` unifies with `Option<T2>`, the engine must also unify `T1` with `T2`. This structural decomposition is called congruence, and it is the caller's responsibility to implement it by recursively calling `unify_var_var` on corresponding type arguments whenever the top-level type constructors match. ena provides the primitives; the congruence rule is encoded in the inference engine's `unify(ty1, ty2)` function that pattern-matches on type structure.

**Persistent strategy for functional solvers.** `Persistent<K>` uses a persistent (immutable, structurally shared) vector as its backing store. Creating a snapshot is O(1) via pointer copy; rollback discards the copy. This strategy is appropriate for solvers that want to explore multiple branches simultaneously without sequential rollback — for example, a Prolog-style resolution engine or a type-class instance search that needs to fork and rejoin inference states.

**Ecosystem consumers.** ena is used by:

- [Chalk](https://github.com/rust-lang/chalk): rustc's trait solver, which unifies type variables during impl selection. The solver issues queries of the form "does `Vec<T>: Iterator`?" and uses ena's union-find to track `T`'s binding constraints across the goal tree.
- [Polonius](https://github.com/rust-lang/polonius): the next-generation Rust borrow checker. Polonius reformulates borrow checking as Datalog constraint solving over loan/region variables; ena tracks which region variables have been equated by the lifetime inference algorithm.
- [Salsa](https://github.com/salsa-rs/salsa): the incremental computation framework used by rust-analyzer. Salsa's dependency tracking uses union-find to merge equivalent query cells, enabling it to avoid re-running queries when inputs change but their observable output is equivalent.
- Chalk's successor, the *next-generation trait solver* being integrated directly into rustc (`rustc_trait_selection`): it also uses ena for type variable binding during the recursive goal search.

Any Rust compiler or type-checker project that needs Hindley-Milner or bidirectional type inference should reach for ena before rolling a custom union-find implementation. The snapshot/rollback feature alone justifies the dependency for any solver that does speculative backtracking.

## 178.5 GPU and ML from Rust

### rust-cuda: Device-Side Rust for NVIDIA GPUs

[rust-cuda](https://github.com/rust-gpu/rust-cuda) enables writing CUDA device-side code in Rust. The Rust compiler supports `--target nvptx64-nvidia-cuda` as a tier 2 target; rust-cuda provides the runtime layer, attribute macros, and safe device-memory abstractions that make this target usable in a real project.

**Compilation model.** Device crates are compiled as:

```bash
cargo build --target nvptx64-nvidia-cuda --crate-type cdylib
```

This produces a `.ptx` text file (PTX assembly). The host-side runtime loads it via `cuModuleLoadData` and launches kernels by name using `cuLaunchKernel`. The `rustc_codegen_nvvm` backend (an alternative to the default LLVM PTX backend) instead produces NVVM IR — an LLVM IR subset — for `libnvvm`, NVIDIA's own optimization and PTX-generation library. NVVM tends to produce better PTX than the LLVM PTX backend for certain patterns because it has access to NVIDIA-proprietary optimizations.

**Kernel authoring.** Device entry points are marked with `#[kernel]`; the surrounding framework provides safe wrappers for built-in CUDA thread indices and device memory:

```rust
use rust_cuda::kernel;

#[kernel::kernel]
#[allow(improper_ctypes_definitions)]
pub fn vector_add(
    a:   &[f32],
    b:   &[f32],
    c:   &mut [f32],
) {
    let tid = rust_cuda::device::thread::ThreadIdx::x() as usize
            + rust_cuda::device::block::BlockIdx::x() as usize
            * rust_cuda::device::block::BlockDim::x() as usize;
    if tid < c.len() {
        c[tid] = a[tid] + b[tid];
    }
}
```

The `#[kernel]` macro generates both a device-side entry point (compiled to PTX) and a host-side launcher that handles the `cuLaunchKernel` call, argument marshalling, and error checking.

**Device-side constraints.** The device target is `core`-only: no `std`, no heap allocator, no unwinding. The `alloc` crate is not available because CUDA device kernels have no general heap. Closures, iterator adapters, and generic functions compile correctly as long as they fully monomorphize to code the PTX backend can lower. `f16` and `bf16` require feature-gating. SIMD requires target-feature flags for the specific CUDA compute capability.

**Comparison with CUDA C++.** The primary advantage of rust-cuda is the `RustToCuda` trait, which allows sharing data-structure definitions (enums, structs, bitfields) between host Rust code and device Rust code without unsafe pointer casts. In CUDA C++ the device and host sides of a struct must be manually kept in sync; `RustToCuda` derives the device-side layout from the host-side Rust type automatically. Slice bounds are passed alongside the pointer, so the device-side `&[f32]` is a fat pointer (ptr + len) rather than a raw `float *`, enabling safe bounds-checked indexing on the device without a separate `size` argument.

The primary disadvantage is maturity: the CUDA ecosystem's tooling (Nsight, nvcc, thrust, cublas) all assume C++ and interoperate poorly with externally-compiled PTX. CUDA's parallel thread execution (PTX) ISA is stable across GPU generations, so the PTX produced by rust-cuda runs correctly — but performance profiling, warp occupancy analysis, and memory access pattern visualization require Nsight Compute, which expects CUDA C++ source. Engineers who need tight integration with CUDA libraries (cuDNN, cuBLAS, NCCL) will find the interop story awkward: calling a cuBLAS routine from a rust-cuda host program requires `unsafe` FFI calls through raw C bindings, exactly as from any other language. For compute-intensive custom kernels where the safety and type-sharing benefits of Rust are worth the tooling penalty, rust-cuda is a compelling choice.

### Burn: A Pluggable Deep Learning Framework

[Burn](https://burn.dev) is a deep learning framework whose central design goal is backend portability. A model written in Burn compiles to any supported backend by changing a type parameter; the model code itself is unchanged. Supported backends include WGPU (cross-platform GPU via Vulkan/Metal/DX12), CUDA (via cuBLAS and custom CubeCL kernels), ROCm (via hipBLAS), `ndarray` (CPU, pure Rust), and `candle` (another pure-Rust ML backend).

```toml
[package]
name    = "burn-example"
version = "0.1.0"
edition = "2024"

[dependencies]
burn = { version = "0.15", features = ["wgpu", "train"] }
```

**The `Module` trait and `Tensor` type.** Neural network layers implement `Module<B>` where `B: Backend`. The derive macro `#[derive(Module)]` generates the forward/backward hooks, parameter iteration for the optimizer, and serialization for checkpointing. The `Tensor<B, D, K>` type carries the backend `B`, tensor rank `D` (a `const` generic), and kind `K` (`Float`, `Int`, or `Bool`). Rank mismatches detectable statically — passing a 3D tensor to a function expecting 2D — are caught at compile time with a clear error.

```rust
use burn::{
    module::Module,
    nn::{Linear, LinearConfig, Relu},
    tensor::{backend::Backend, Tensor},
};

#[derive(Module, Debug)]
pub struct Mlp<B: Backend> {
    linear1:    Linear<B>,
    linear2:    Linear<B>,
    activation: Relu,
}

impl<B: Backend> Mlp<B> {
    pub fn new(device: &B::Device, d_in: usize, d_hidden: usize, d_out: usize) -> Self {
        Self {
            linear1:    LinearConfig::new(d_in, d_hidden).init(device),
            linear2:    LinearConfig::new(d_hidden, d_out).init(device),
            activation: Relu::new(),
        }
    }

    pub fn forward(&self, x: Tensor<B, 2>) -> Tensor<B, 2> {
        let h = self.linear1.forward(x);
        let h = self.activation.forward(h);
        self.linear2.forward(h)
    }
}
```

Switching from WGPU to CUDA requires only changing the backend type at the call site; the model struct and its `forward` method are identical.

**Checkpointing and inference.** `#[derive(Module)]` generates serialization and deserialization hooks compatible with Burn's record format. Saving a checkpoint is one line; loading for inference on a different backend (e.g., training on CUDA, inference on CPU `ndarray`) works through the same `Module::load_record` API because the backend is a type parameter — the loaded weights are simply reinterpreted for the new backend's tensor type:

```rust
use burn::record::{CompactRecorder, Recorder};

let recorder = CompactRecorder::new();
recorder.record(model.into_record(), "/tmp/model_checkpoint".into()).unwrap();

let cpu_model: Mlp<ndarray::NdArray> = Mlp::new(&Default::default(), 784, 256, 10);
let record = CompactRecorder::new()
    .load("/tmp/model_checkpoint".into(), &Default::default())
    .unwrap();
let cpu_model = cpu_model.load_record(record);
```

The `into_record()` / `load_record()` abstraction is generated by `#[derive(Module)]` and walks the module tree collecting all `Param<Tensor<B, D>>` fields.

**Training loop.** `LearnerBuilder` assembles optimizer, loss function, learning rate scheduler, and data loaders:

```rust
use burn::optim::AdamConfig;
use burn::train::LearnerBuilder;

let learner = LearnerBuilder::new("/tmp/artifacts")
    .with_file_checkpointer(10)
    .devices(vec![device])
    .num_epochs(20)
    .build(model, AdamConfig::new().init(), 1e-3);

learner.fit(train_loader, valid_loader);
```

Burn's automatic differentiation backend wraps any primal backend with a tape-based AD engine (no source transformation, no JIT compilation of gradients). Switching from inference to training mode is a type-level change: replace `B: Backend` with `B: AutodiffBackend` and call `tensor.backward()` on the loss.

### CubeCL: GPU Kernel DSL via Proc-Macros

[CubeCL](https://burn.dev/books/cubecl/) is the kernel-authoring layer that Burn uses internally for its GPU backends and exposes publicly for custom GPU programming. It is not a binding to CUDA or Metal: it is a DSL embedded in Rust via proc-macros that compiles kernel functions to WGSL (WGPU backend), MSL (Metal backend), or CUDA PTX (CUDA backend) from a single source. The compilation is JIT, driven at runtime by the CubeCL runtime when a kernel is first launched with a given set of `#[comptime]` parameter values.

```toml
[package]
name    = "cubecl-example"
version = "0.1.0"
edition = "2024"

[dependencies]
cubecl      = { version = "0.4", features = ["wgpu"] }
cubecl-wgpu = "0.4"
```

**The `#[cube]` proc-macro.** A function annotated with `#[cube]` is *not* compiled as ordinary Rust MIR by the Rust compiler. Instead, the proc-macro walks the token stream and produces a `KernelSource` representation: a typed IR that CubeCL's runtime JIT-compiles to the target language when the kernel is first launched. The Rust compiler still type-checks the function body against the `CubePrimitive` and `Array<T>` types provided by CubeCL, but the code generation path bypasses MIR entirely.

```rust
use cubecl::prelude::*;

#[cube(launch)]
fn vector_add<F: Float>(
    a:          &Array<F>,
    b:          &Array<F>,
    out:        &mut Array<F>,
    #[comptime] len: u32,
) {
    let i = ABSOLUTE_POS;
    if i < len {
        out[i] = a[i] + b[i];
    }
}
```

**`#[comptime]` parameters and specialization.** Parameters marked `#[comptime]` are *specialization constants* — their values are baked into the JIT-compiled kernel source at launch time. This means the `if i < len` comparison, where `len` is comptime-known, is emitted as a comparison with a literal constant rather than a load from a buffer. The GPU compiler can then:

- Unroll inner loops bounded by `len` if it is small.
- Eliminate the bounds check entirely if the grid is sized to guarantee `i < len`.
- Specialize reduction kernels for power-of-two block sizes.

CubeCL caches compiled kernels keyed by `(backend, kernel_function_id, comptime_values)`. Repeated launches with the same `len` pay zero JIT cost after the first; launches with a different `len` trigger a new JIT compilation and cache insertion.

**Launching a kernel.** The `#[cube(launch)]` attribute variant generates a `vector_add::launch` free function alongside the kernel body. Launch parameters include `CubeCount` (grid dimensions in cubes), `CubeDim` (block dimensions), and the runtime arguments followed by `#[comptime]` values:

```rust
use cubecl_wgpu::{WgpuDevice, WgpuRuntime};
use cubecl::prelude::*;

fn run_add(n: usize) {
    let device = WgpuDevice::default();
    let client = cubecl_wgpu::init_sync::<WgpuRuntime>(&device, Default::default());

    let a_data: Vec<f32> = (0..n).map(|i| i as f32).collect();
    let b_data: Vec<f32> = (0..n).map(|i| i as f32 * 2.0).collect();

    let a   = client.tensor_uninitialized(vec![n]);
    let b   = client.tensor_uninitialized(vec![n]);
    let out = client.tensor_uninitialized(vec![n]);

    client.copy_from_host(a_data.as_slice().into(), &a);
    client.copy_from_host(b_data.as_slice().into(), &b);

    let cube_dim   = CubeDim::new(256, 1, 1);
    let cube_count = CubeCount::Static(n.div_ceil(256) as u32, 1, 1);

    unsafe {
        vector_add::launch::<f32, WgpuRuntime>(
            &client, cube_count, cube_dim,
            ArrayArg::from_raw_parts(&a, n, 1),
            ArrayArg::from_raw_parts(&b, n, 1),
            ArrayArg::from_raw_parts(&out, n, 1),
            n as u32,
        )
    };
}
```

The final `n as u32` argument corresponds to `#[comptime] len` in the kernel signature — it is embedded as a literal in the JIT-compiled WGSL, not passed as a buffer uniform.

**Supported primitives and built-in IDs.** `CubePrimitive` is implemented by `f32`, `f64`, `f16` (feature-gated), `bf16` (feature-gated), `i32`, `i64`, `u32`, `u64`, and `bool`. The built-in identifiers map to their CUDA/WGSL equivalents:

| CubeCL ID | CUDA equivalent | WGSL equivalent |
|-----------|----------------|-----------------|
| `ABSOLUTE_POS` | `blockIdx.x * blockDim.x + threadIdx.x` | `global_invocation_id.x` |
| `UNIT_POS` | `threadIdx.x` | `local_invocation_id.x` |
| `CUBE_POS` | `blockIdx.x` | `workgroup_id.x` |
| `CUBE_DIM` | `blockDim.x` | `workgroup_size.x` |
| `UNIT_POS_GLOBAL` | flattened global thread index | `global_invocation_index` |

Two-dimensional and three-dimensional variants (suffixed `_X`, `_Y`, `_Z`) are also provided for kernels that use 2D or 3D grid layouts.

**The CubeCL kernel IR.** When the `#[cube]` macro processes a function, it lowers the token stream to CubeCL's own typed IR — not Rust MIR, not LLVM IR, not WGSL. This IR is an SSA-like representation with nodes for arithmetic, control flow, array access, and comptime-conditional branching. At JIT time the CubeCL runtime traverses this IR and emits the target language (WGSL, MSL, or CUDA C). The two-phase design (macro-time lowering to kernel IR, runtime JIT to target) is what enables single-source multi-backend: the macro runs once at Rust compile time, producing a `KernelExpansion` struct embedded in the binary; the JIT selects the emitter at runtime based on the active backend.

The comptime-conditional in the kernel IR is distinct from Rust's `if cfg!(...)`: CubeCL's `#[comptime]` values are not compile-time constants from the Rust perspective (they are runtime values of the host program), but they are treated as constants by the JIT. The kernel IR therefore has a `ComptimeNode` variant that the JIT evaluates at kernel-compilation time, inlining the value into the generated source. This is the mechanism behind block-size specialization: the grid/block dimensions are passed as `#[comptime]` values, and the JIT produces a kernel with those dimensions as literal constants rather than loads from a buffer.

**How Burn uses CubeCL for operation fusion.** When Burn's WGPU backend evaluates a sequence of elementwise operations — for example `relu → mul → add` — the `burn-fusion` module intercepts the operation stream before any kernel is launched. It detects fusable sequences (elementwise ops with the same output shape that share input tensors) and emits a single fused CubeCL kernel rather than three separate kernel launches. The fused kernel is parameterized by the elementwise function sequence, which is encoded as a `#[comptime]` parameter (a small enum or type-level list), enabling the JIT to inline the sequence as a straight-line computation. Intermediate activations never materialize in GPU memory, eliminating the memory-bandwidth bottleneck that otherwise dominates small-operation latency on modern GPUs. This fusion pass is how Burn achieves inference throughput comparable to hand-tuned backends despite its high-level API.

## Chapter Summary

- **llvm-sys** provides mechanical `bindgen`-generated bindings to the LLVM C API. Edition 2024 requires `unsafe extern "C" { ... }` for FFI declarations, making unsafety visible at declaration sites. Version selection uses matching crate version (`"220"`) and feature flag (`"llvm22-0"`); `LLVM_SYS_220_PREFIX` overrides automatic `llvm-config` discovery.

- **inkwell** wraps llvm-sys in safe Rust by threading a `'ctx` phantom lifetime through all IR value types (`FunctionValue<'ctx>`, `IntValue<'ctx>`, `BasicBlock<'ctx>`). This prevents use-after-context-drop at compile time with zero runtime overhead. JIT execution via `ExecutionEngine` and `JitFunction<'ctx, F>` follows the same lifetime discipline. The NPM is exposed through `module.run_passes(pipeline_string, &target_machine, opts)`.

- **iron-kaleidoscope** is the best available worked example of a complete LLVM-based compiler in Rust using inkwell. It targets an older inkwell API (pre-0.4) and requires porting to current inkwell; the structural lessons about separating lexer, parser, codegen context, and JIT lifecycle apply directly to production compilers.

- **melior** binds to the MLIR C API with the same `'ctx`-lifetime discipline as inkwell. The two-type model (`OwnedOperation<'c>` vs. `OperationRef<'c, '_>`) reflects MLIR's ownership rules, preventing double-free of block-owned operations. The `ods-dialects` feature generates typed builders for standard dialects; `PassManager::add_pipeline` accepts the same string as `mlir-opt -passes=`.

- **pliron** reimplements MLIR's structural model natively in Rust via `Op`/`Type`/`Attribute` traits, arena allocation, and `TypeId`-based downcasting — no C++ dependency, no MLIR wire compatibility. The right choice for new compiler projects that want MLIR-style discipline without an LLVM/MLIR installation requirement.

- **Calyx** is the leading IR for verified hardware generation from MLIR-level computation. Its explicit separation of structural (cells, wires) and behavioral (control schedule) enables correctness proofs unavailable in raw HLS. It lowers to synthesizable SystemVerilog via CIRCT's `calyx-to-hw` pass.

- **object** provides a unified zero-copy read/write API for ELF, Mach-O, PE/COFF, and WebAssembly. `File::parse(&[u8])` detects format from magic bytes; the `Object` trait provides sections, symbols, exports, and relocations through a format-agnostic interface. Used by rustc's self-contained linker and `cargo-binutils`.

- **gimli** is a lazy, zero-copy DWARF 5 reader/writer. Abbreviation tables are memoized per CU; all other parsing is on-demand. The `write` module is used by rustc to emit split-DWARF `.dwo` files. Performance is optimal for symbolication tools that touch only a small fraction of a binary's debug information.

- **addr2line** builds on gimli to resolve virtual addresses to `(file, line, column, function)` tuples in O(log n) after O(n) construction. It handles inlined call chains, returning multiple frames per PC. Used by the Rust `backtrace` crate for panic symbolication.

- **ena** implements union-find with value association and O(1) snapshot/rollback, extracted from rustc's type inference engine. `unify_var_var`, `unify_var_value`, `probe_value`, `snapshot`, and `rollback_to` are the core API. Used by Chalk (trait solving), Polonius (borrow checking), and Salsa (incremental computation).

- **rust-cuda** compiles device-side Rust to PTX via `--target nvptx64-nvidia-cuda`. The device subset is `core`-only with no heap allocation. `RustToCuda` shares type definitions across host and device. Status is experimental; the ecosystem assumes CUDA C++ for tooling.

- **Burn** is a backend-portable deep learning framework where `B: Backend` is a type parameter throughout. `#[derive(Module)]` generates training/inference/checkpoint infrastructure. Switching backends requires only changing a type alias. Burn uses CubeCL for its GPU kernel layer.

- **CubeCL** is a GPU kernel DSL embedded via proc-macros targeting CUDA, Metal, and WGPU from one kernel definition. `#[comptime]` parameters are baked into JIT-compiled kernel source as constants, enabling loop unrolling and specialization. Burn uses CubeCL's comptime-parameterized fused kernels to eliminate intermediate GPU allocations in elementwise operation sequences.
