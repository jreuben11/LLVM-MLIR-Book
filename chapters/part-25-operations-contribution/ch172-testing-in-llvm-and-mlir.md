# Chapter 172 — Testing in LLVM and MLIR

*Part XXV — Operations, Bindings, and Contribution*

A compiler that cannot be tested cannot be trusted. LLVM's test infrastructure reflects decades of accumulated wisdom about what kinds of tests catch what kinds of bugs: regression tests that prevent known-fixed bugs from re-emerging, unit tests that verify individual data structures and algorithms in isolation, differential tests that compare two executions for agreement, and fuzz tests that find bugs no human would think to write a test for. MLIR inherits this infrastructure and adds idioms suited to its dialect-based architecture. This chapter covers every layer of LLVM and MLIR's testing story — lit, FileCheck, Google Test, MLIR-specific patterns, and fuzzing — with enough practical detail to write, run, and debug tests effectively.

---

## 172.1 Test Infrastructure Overview

LLVM uses two complementary test frameworks:

**lit + FileCheck**: for end-to-end tests that verify the output of compiler tools. A test is a source file containing embedded shell commands; the output is matched against patterns using FileCheck. These tests are fast, language-agnostic, and cover the most important correctness property: does the tool produce the expected output?

**Google Test**: for unit tests of internal C++ APIs. These tests compile and run directly against LLVM's libraries, verifying that individual functions, classes, and data structures behave correctly in isolation.

Directory layout:
```
llvm/
  test/              # lit + FileCheck regression tests
    Transforms/      # opt pass tests
    Analysis/        # analysis pass tests
    CodeGen/         # instruction selection, regalloc tests
    MC/              # MC layer tests
    IR/              # IR construction, parsing tests
    tools/           # clang, llc, llvm-dis, etc.
  unittests/         # Google Test unit tests
    ADT/             # ArrayRef, StringRef, DenseMap, etc.
    Analysis/        # alias analysis, dominance, etc.
    IR/              # Module, Function, IRBuilder, etc.
    Support/         # file system, error handling, etc.

mlir/
  test/              # lit + FileCheck regression tests
    Dialect/         # per-dialect tests
    Transforms/      # transformation tests
    Conversion/      # dialect conversion tests
    Integration/     # integration tests (run via mlir-cpu-runner)
  unittests/         # Google Test unit tests
```

## 172.2 lit (LLVM Integrated Tester)

### What lit Is

`lit` is a test runner written in Python that discovers and executes test files. It reads embedded shell commands from test files, runs them, and checks the return codes. `FileCheck` is a separate tool that `lit`-invoked commands pipe into; it verifies output patterns.

### Running Tests

```bash
# Run all tests in a directory
llvm-lit llvm/test/Transforms/InstCombine/

# Run a single test
llvm-lit llvm/test/Transforms/InstCombine/add.ll

# Run with verbose output
llvm-lit -v llvm/test/Transforms/InstCombine/add.ll

# Run with timing information
llvm-lit --time-tests llvm/test/Transforms/

# Show tests that are expected to fail (XFAIL) or unsupported (UNSUPPORTED)
llvm-lit --show-unsupported --show-xfail llvm/test/

# Run specific tests matching a glob pattern
llvm-lit llvm/test/Transforms/InstCombine/ --filter='.*add.*'

# Limit parallelism
llvm-lit -j4 llvm/test/
```

After building with CMake/Ninja, the `check-llvm` target runs all lit tests:
```bash
ninja check-llvm
ninja check-mlir
ninja check-all   # all sub-projects
```

### Test File Structure

A typical lit test file (`llvm/test/Transforms/InstCombine/add.ll`):

```llvm
; NOTE: Assertions have been autogenerated by utils/update_test_checks.py
; RUN: opt -passes=instcombine -S < %s | FileCheck %s

define i32 @add_zero(i32 %x) {
; CHECK-LABEL: @add_zero(
; CHECK-NEXT:    ret i32 [[X:%.*]]
  %r = add i32 %x, 0
  ret i32 %r
}

define i32 @add_negative(i32 %x) {
; CHECK-LABEL: @add_negative(
; CHECK-NEXT:    [[R:%.*]] = add i32 [[X:%.*]], -1
; CHECK-NEXT:    ret i32 [[R]]
  %r = add i32 %x, -1
  ret i32 %r
}
```

Key substitutions in `RUN:` lines:
- `%s`: the test file itself (absolute path).
- `%t`: a temporary file; `%t.ll`, `%t.o`, etc. for derivative files.
- `%S`: the directory containing the test file.
- `%p`: same as `%S`.
- `%T`: the directory for temporary files.
- `%{python}`: the Python interpreter.

### lit Configuration Files

`lit.cfg.py` (in each test directory) defines substitutions and features:

```python
# llvm/test/lit.cfg.py (simplified)
import lit.formats
config.test_format = lit.formats.ShTest()

# Define tool substitutions
config.substitutions.append(
    ('%FileCheck', os.path.join(config.llvm_tools_dir, 'FileCheck')))
config.substitutions.append(
    ('%opt', os.path.join(config.llvm_tools_dir, 'opt')))

# Define available features
if config.have_zlib:
    config.available_features.add('zlib')
config.available_features.add('asserts' if config.enable_assertions else 'no-asserts')
```

`lit.site.cfg.py` is generated by CMake and contains build-specific paths. The structure: `lit.site.cfg.py` sets `config.llvm_tools_dir`, then `lit.cfg.py` constructs substitutions from it.

### XFAIL and UNSUPPORTED

```llvm
; XFAIL: *                    ; expected to fail on all platforms
; XFAIL: target=x86_64-*     ; expected to fail on x86-64 only
; REQUIRES: asserts           ; only run when LLVM_ENABLE_ASSERTIONS=ON
; REQUIRES: x86-registered-target
; UNSUPPORTED: windows        ; skip on Windows
```

`XFAIL` marks a known failure — the test is expected to fail and is not reported as an error. `REQUIRES` marks a conditional test — only run if the feature is present. `UNSUPPORTED` skips the test entirely.

## 172.3 FileCheck

FileCheck is a pattern-matching tool for verifying command output. It reads patterns from the test file and matches them against the tool's standard output.

### Basic Directives

```llvm
; CHECK: pattern        ; Match a line containing "pattern" (anywhere after the previous CHECK)
; CHECK-NEXT: pattern   ; Match the VERY NEXT line (no blank lines allowed)
; CHECK-SAME: pattern   ; Match pattern on the SAME line as the previous CHECK
; CHECK-NOT: pattern    ; Assert "pattern" does NOT appear between previous and next CHECK
; CHECK-EMPTY:          ; Assert the next line is empty
; CHECK-LABEL: pattern  ; Like CHECK, but also resets the search position (for anchoring)
```

### CHECK-LABEL for Functions

`CHECK-LABEL` is used to anchor checks to the start of a function. This prevents patterns from matching in the wrong function:

```llvm
define i32 @foo(i32 %x) {
; CHECK-LABEL: @foo(
; CHECK-NEXT:    ret i32 0
  ret i32 0
}

define i32 @bar(i32 %x) {
; CHECK-LABEL: @bar(
; CHECK-NEXT:    ret i32 %x
  ret i32 %x
}
```

Without `CHECK-LABEL`, a `CHECK` in `@foo` could match output from `@bar`.

### Variable Capture and Reuse

```llvm
; Capture a name into [[REG]] and reuse it
; CHECK: [[REG:%[a-zA-Z0-9_.]+]] = alloca i32
; CHECK: store i32 42, ptr [[REG]]
; CHECK: [[VAL:%[a-zA-Z0-9_.]+]] = load i32, ptr [[REG]]
; CHECK: ret i32 [[VAL]]
```

The `[[VAR:regex]]` syntax captures the text matching `regex` into `VAR`; subsequent `[[VAR]]` references match the same text. Variable names are case-sensitive.

### CHECK-DAG: Order-Independent Matching

```llvm
; CHECK-DAG: %a = add i32 %x, 1
; CHECK-DAG: %b = add i32 %y, 1
; The two adds may appear in any order
```

`CHECK-DAG` directives match in any order between `CHECK-LABEL` or the file boundaries. Useful for instruction selection outputs where instruction order is not deterministic.

### Multiple Check Prefixes

```bash
# RUN: opt -passes=instcombine %s -S | FileCheck %s --check-prefixes=CHECK,INSTCOMBINE
# RUN: opt -passes=aggressive-instcombine %s -S | FileCheck %s --check-prefixes=CHECK,AGGRESSIVE
```

```llvm
; CHECK-LABEL: @test(
; CHECK:         ret
; INSTCOMBINE:   ret i32 0
; AGGRESSIVE:    ret i32 42
```

Multiple prefixes allow a single test file to test multiple tools or configurations.

### update_test_checks.py

Writing `CHECK` patterns by hand is tedious and error-prone. LLVM provides `utils/update_test_checks.py` to autogenerate them:

```bash
# Autogenerate CHECK lines for an InstCombine test
python3 llvm/utils/update_test_checks.py \
  --tool-binary=build/bin/opt \
  llvm/test/Transforms/InstCombine/add.ll

# For MLIR tests
python3 mlir/utils/update_mlir_test_checks.py \
  --tool-binary=build/bin/mlir-opt \
  mlir/test/Transforms/canonicalize.mlir
```

This runs the tool, captures the output, and inserts the `CHECK` lines that match it. The generated patterns use `[[...]]` variable captures for SSA names. This workflow:
1. Write the test function without `CHECK` lines.
2. Run `update_test_checks.py` to generate the checks.
3. Review the generated checks to ensure they actually test what you intend (not just the current output).
4. Commit.

## 172.4 Unit Tests

### Google Test Structure

LLVM unit tests use Google Test (`llvm/unittests/`):

```cpp
// llvm/unittests/ADT/ArrayRefTest.cpp
#include "llvm/ADT/ArrayRef.h"
#include "gtest/gtest.h"

using namespace llvm;

TEST(ArrayRefTest, EmptyArray) {
  ArrayRef<int> A;
  EXPECT_TRUE(A.empty());
  EXPECT_EQ(0u, A.size());
}

TEST(ArrayRefTest, FromVector) {
  std::vector<int> V = {1, 2, 3};
  ArrayRef<int> A = V;
  EXPECT_EQ(3u, A.size());
  EXPECT_EQ(2, A[1]);
  ASSERT_FALSE(A.empty());
}
```

Key assertions:
- `EXPECT_EQ(a, b)`: non-fatal; test continues on failure.
- `ASSERT_EQ(a, b)`: fatal; test stops on failure.
- `EXPECT_TRUE(x)`, `EXPECT_FALSE(x)`.
- `EXPECT_NE(a, b)`, `EXPECT_LT(a, b)`, `EXPECT_LE(a, b)`.
- `EXPECT_STREQ(a, b)`: C-string comparison.
- `EXPECT_THAT(container, Contains(x))`: Matcher-based assertions (Google Mock).

### Running Unit Tests

```bash
# Build and run all unit tests
ninja llvm-unit-tests
./bin/LLVMSupportTests
./bin/LLVMAnalysisTests
./bin/LLVMADTTests

# Run a specific test suite
./bin/LLVMADTTests --gtest_filter="ArrayRefTest.*"

# Run a specific test
./bin/LLVMADTTests --gtest_filter="ArrayRefTest.EmptyArray"

# List all tests in a binary
./bin/LLVMADTTests --gtest_list_tests
```

### Writing a New Unit Test

```cpp
// llvm/unittests/IR/MyPassTest.cpp
#include "llvm/IR/IRBuilder.h"
#include "llvm/IR/LLVMContext.h"
#include "llvm/IR/Module.h"
#include "llvm/IR/PassManager.h"
#include "llvm/Transforms/Scalar/MyPass.h"
#include "gtest/gtest.h"

using namespace llvm;

class MyPassTest : public testing::Test {
protected:
  LLVMContext Ctx;
  
  // Helper: build a module with a simple function
  std::unique_ptr<Module> buildModule() {
    auto M = std::make_unique<Module>("test", Ctx);
    IRBuilder<> B(Ctx);
    auto *FT = FunctionType::get(B.getInt32Ty(), {B.getInt32Ty()}, false);
    auto *F = Function::Create(FT, GlobalValue::ExternalLinkage, "test", M.get());
    auto *BB = BasicBlock::Create(Ctx, "entry", F);
    B.SetInsertPoint(BB);
    B.CreateRet(F->getArg(0));
    return M;
  }
};

TEST_F(MyPassTest, IdentityFunction) {
  auto M = buildModule();
  FunctionPassManager FPM;
  FPM.addPass(MyPass());
  FunctionAnalysisManager FAM;
  // Run the pass
  FPM.run(*M->getFunction("test"), FAM);
  // Verify expected changes
  EXPECT_EQ(M->getFunction("test")->size(), 1u);  // still one BB
}
```

## 172.5 MLIR-Specific Testing

### mlir-opt as Test Driver

In MLIR, `mlir-opt` is the universal test driver. Tests use the same lit + FileCheck framework:

```mlir
// mlir/test/Transforms/canonicalize.mlir
// RUN: mlir-opt %s --canonicalize | FileCheck %s

func.func @fold_const_add() -> i32 {
  // CHECK-LABEL: func.func @fold_const_add
  // CHECK-NEXT:    %[[C:.*]] = arith.constant 3 : i32
  // CHECK-NEXT:    return %[[C]]
  %a = arith.constant 1 : i32
  %b = arith.constant 2 : i32
  %r = arith.addi %a, %b : i32
  return %r : i32
}
```

MLIR-specific `mlir-opt` flags for debugging tests:
```bash
# Print IR before and after every pass
mlir-opt --mlir-print-ir-before-all --mlir-print-ir-after-all input.mlir

# Print IR only when changed
mlir-opt --mlir-print-ir-after-change input.mlir

# Print IR after specific pass
mlir-opt --mlir-print-ir-after=canonicalize input.mlir

# Verify after every pass (slower but catches bugs earlier)
mlir-opt --mlir-verify-each input.mlir
```

### Testing Dialect Conversions

Conversion tests verify that a dialect lowering produces the expected output:

```mlir
// RUN: mlir-opt %s --convert-linalg-to-loops | FileCheck %s

func.func @matmul(%A: memref<4x4xf32>, %B: memref<4x4xf32>, %C: memref<4x4xf32>) {
  // CHECK-LABEL: func.func @matmul
  // CHECK:         scf.for %[[I:.*]] = {{.*}} to {{.*}}
  // CHECK:           scf.for %[[J:.*]] = {{.*}} to {{.*}}
  // CHECK:             scf.for %[[K:.*]] = {{.*}} to {{.*}}
  linalg.matmul ins(%A, %B : memref<4x4xf32>, memref<4x4xf32>)
                outs(%C : memref<4x4xf32>)
  return
}
```

### Testing with mlir-cpu-runner

Integration tests that run the compiled program use `mlir-cpu-runner` (or `mlir-runner`) as a JIT executor:

```mlir
// RUN: mlir-cpu-runner %s -e main -entry-point-result=void \
// RUN:   --shared-libs=%mlir_lib_dir/libmlir_runner_utils%shlibext | FileCheck %s

func.func @main() {
  %c42 = arith.constant 42 : i32
  vector.print %c42 : i32
  return
  // CHECK: 42
}
```

### MLIR FileCheck Patterns

MLIR IR has structured syntax that benefits from careful pattern design:

```mlir
// Matching types in MLIR (types can be complex)
// CHECK: %[[VAL:.*]] = memref.alloc() : memref<4x4xf32, 1>

// Matching attributes
// CHECK: linalg.generic {indexing_maps = [#map0, #map1, #map2]

// Matching dialect attributes
// CHECK: func.func @{{.*}}(%[[ARG0:.*]]: i32 {llvm.noalias})

// Using CHECK-DAG for reordered outputs (e.g., function attributes)
// CHECK-DAG: llvm.func @malloc
// CHECK-DAG: llvm.func @free
```

## 172.6 Fuzzing the Compiler

### llvm-stress: Random IR Generation

`llvm-stress` generates random LLVM IR programs:

```bash
# Generate a random module with 1000 operations, seed 42
llvm-stress -size=1000 -seed=42 -o random.ll

# Pipe directly to opt
llvm-stress | opt -passes=instcombine -S
```

The generated IR is syntactically valid but semantically arbitrary — it uses legal LLVM IR constructs with random combinations of types, values, and instructions. Running thousands of random programs through the compiler and checking for crashes is a basic fuzzing strategy.

### llvm-reduce and mlir-reduce

When a crash is found, the IR is often large. `llvm-reduce` (and `mlir-reduce` for MLIR) reduces the IR to a minimal reproducer using delta debugging:

```bash
# llvm-reduce requires an "interesting" script that returns 0 when the bug is present
cat > interesting.sh << 'EOF'
#!/bin/bash
opt -passes=instcombine "$1" &>/dev/null
exit $?   # Invert: we want crashes, so return 0 when the exit code is nonzero
EOF
# Actually for a crash:
cat > interesting.sh << 'EOF'
#!/bin/bash
opt -passes=instcombine "$1" 2>&1 | grep -q "LLVM ERROR" && exit 0
exit 1
EOF

chmod +x interesting.sh
llvm-reduce --test=./interesting.sh --test-arg=%s crashing.ll
# Output: reduced.ll (minimal crashing input)
```

For MLIR:
```bash
mlir-reduce --test=./interesting.sh crashing.mlir
```

`llvm-reduce` applies a sequence of delta-reduction strategies:
- Remove functions, basic blocks, instructions.
- Replace values with `undef` or constants.
- Simplify types.

### clang-fuzzer and llvm-opt-fuzzer

LLVM participates in OSS-Fuzz, Google's continuous fuzzing infrastructure:

```bash
# Build clang with AddressSanitizer and fuzzer support
cmake -DLLVM_USE_SANITIZER=Address -DLLVM_BUILD_RUNTIME=OFF ...
ninja clang-fuzzer

# Run the fuzzer
./bin/clang-fuzzer corpus_dir/ -max_len=1024 -jobs=8
```

Key fuzzers:
- **`clang-fuzzer`**: fuzzes C/C++ parsing and type-checking. Inputs are C source files.
- **`llvm-opt-fuzzer`**: fuzzes opt passes. Inputs are LLVM IR. The target pass is specified at build time.
- **`llvm-mc-fuzzer`**: fuzzes the MC layer (assembler/disassembler).
- **`mlir-opt-fuzzer`**: fuzzes MLIR optimization passes.

OSS-Fuzz runs these fuzzers continuously and files bugs in LLVM's issue tracker. Historically, tens of thousands of crashes have been found and fixed this way.

### creduce/cvise for C Source Reduction

When a miscompilation is found in C source (not IR), `creduce` or `cvise` reduces the C program:

```bash
# cvise is the actively maintained successor to creduce
cat > interesting.sh << 'EOF'
#!/bin/bash
clang -O2 "$1" -o /tmp/test_binary 2>&1 | grep -q "segfault" && exit 0
/tmp/test_binary && exit 1   # Expected to crash; returns 0 if it does
exit 1
EOF

cvise ./interesting.sh large_crashing.c
# Output: small.c (minimal reproducer)
```

`cvise` is essential for filing actionable bug reports: a 1000-line reproducer is difficult to analyze; a 5-line reproducer allows immediate investigation.

## Chapter Summary

- **lit** (LLVM Integrated Tester) runs test files containing embedded `RUN:` shell commands; `FileCheck` verifies the output against embedded patterns.
- `CHECK`, `CHECK-NEXT`, `CHECK-NOT`, `CHECK-SAME`, `CHECK-LABEL`, `CHECK-DAG` are the core FileCheck directives; variable capture (`[[VAR:regex]]`) enables referencing matched names.
- `update_test_checks.py` automates the generation of `CHECK` patterns from actual tool output; always review generated checks to ensure they test the right behavior.
- **Google Test** (`llvm/unittests/`) provides unit tests for LLVM's internal C++ APIs; `TEST_F` fixtures enable shared setup; `EXPECT_*`/`ASSERT_*` macros assert correctness conditions.
- **MLIR tests** use `mlir-opt` as the universal test driver; `--mlir-print-ir-before-all`, `--mlir-verify-each`, and `--debug-only=greedy-rewriter` aid debugging.
- `mlir-cpu-runner` enables end-to-end integration tests that JIT-compile and execute MLIR programs.
- **`llvm-stress`** generates random LLVM IR for crash testing; **`llvm-reduce`**/**`mlir-reduce`** minimize crashing inputs using delta debugging.
- OSS-Fuzz continuously runs `clang-fuzzer`, `llvm-opt-fuzzer`, and `mlir-opt-fuzzer`; historical runs have found tens of thousands of bugs.
- **`cvise`/`creduce`** reduce large C/C++ reproducers to minimal test cases for bug reporting.
