# Chapter 83 — The Target Description

*Part XIV — The Backend*

The LLVM target description is the C++ infrastructure that connects the TableGen-generated tables to the backend optimization and code generation passes. It consists of three main C++ classes — `TargetMachine`, `TargetSubtargetInfo`, and `TargetLowering` — plus the register class infrastructure, instruction info tables, calling convention machinery, and scheduling models. This chapter covers each component, its role in the backend pipeline, and the key APIs that pass authors interact with.

## 83.1 TargetMachine

### 83.1.1 Overview

`TargetMachine` (in `llvm/Target/TargetMachine.h`) is the root of the target description hierarchy. It holds target-wide configuration that does not vary per function or per subtarget feature combination:

```cpp
class TargetMachine {
public:
  // Target triple (e.g., "x86_64-unknown-linux-gnu")
  const Triple &getTargetTriple() const { return TheTriple; }

  // Data layout (pointer size, alignment, endianness):
  const DataLayout &getDataLayout() const { return DL; }

  // Code model (small, kernel, medium, large, tiny):
  CodeModel::Model getCodeModel() const { return CMModel; }

  // Relocation model (PIC, static, dynamic-no-pic):
  Reloc::Model getRelocationModel() const { return RM; }

  // Get subtarget for a specific CPU + features string:
  virtual const TargetSubtargetInfo *
  getSubtargetImpl(const Function &F) const = 0;

  // Create the pass config for this target:
  virtual TargetPassConfig *createPassConfig(PassManagerBase &PM) = 0;
};
```

Each target implements a `TargetMachine` subclass. The subclass holds:
- Subtarget cache (one entry per unique CPU+features combination seen).
- Target-specific data layout string (e.g., `"e-m:e-p270:32:32-p271:32:32-p272:64:64-i64:64-f80:128-n8:16:32:64-S128"` for x86-64 Linux).

```bash
# Dump data layout for a target:
clang -target x86_64-linux-gnu -c -emit-llvm input.c -o - | llvm-dis | head -5
# Outputs: target datalayout = "e-m:e-..."
```

### 83.1.2 Per-Function Subtarget

LLVM supports per-function subtargets: a function can use `__attribute__((target("avx2")))` to request a different feature set from the module default. The target machine creates a different subtarget for that function:

```cpp
// TargetMachine::getSubtargetImpl() implementation:
const X86Subtarget *X86TargetMachine::getSubtargetImpl(const Function &F) const {
  std::string CPU = F.getFnAttribute("target-cpu").getValueAsString();
  std::string Features = F.getFnAttribute("target-features").getValueAsString();
  auto &I = SubtargetMap[CPU + Features];
  if (!I)
    I = std::make_unique<X86Subtarget>(CPU, Features, *this);
  return I.get();
}
```

## 83.2 TargetSubtargetInfo

`TargetSubtargetInfo` (in `llvm/CodeGen/TargetSubtargetInfo.h`) holds CPU-specific information: which features are enabled (AVX2, SVE, etc.), instruction latencies, and register allocation preferences:

```cpp
class TargetSubtargetInfo {
public:
  // Feature flags (generated by -gen-subtarget):
  bool hasAVX2() const { return HasAVX2; }
  bool hasAVX512F() const { return HasAVX512F; }

  // Scheduling model for this CPU:
  const MCSchedModel &getSchedModel() const { return *SchedModel; }

  // Target-specific TTI implementation:
  virtual void getPostRAMutations(
      std::vector<std::unique_ptr<ScheduleDAGMutation>> &) const {}

  // ICache line size (for alignment):
  virtual unsigned getCacheLineSize() const { return 64; }
  virtual unsigned getPrefFunctionAlignment() const { return 4; }
};
```

The subtarget is the key passed to `TargetTransformInfo` (TTI) queries — it provides the data that TTI uses to estimate instruction costs, vectorization profitability, and loop unroll thresholds.

## 83.3 TargetLowering

`TargetLowering` (in `llvm/CodeGen/TargetLowering.h`) is the most important class in the target description. It controls how operations are mapped to SelectionDAG nodes during lowering:

### 83.3.1 Operation and Type Actions

For each pair of (ISD opcode, MVT), `TargetLowering` specifies an **action**:

```cpp
class TargetLowering {
public:
  // Legal action: this operation on this type is natively supported.
  // Custom action: call LowerOperation() to handle it specially.
  // Expand action: expand to a sequence of simpler operations.
  // Promote action: widen the type (e.g., i8 → i32 for operations).
  // LibCall action: replace with a runtime library call.

  LegalizeAction getOperationAction(unsigned Op, EVT VT) const;
  LegalizeAction getTypeAction(LLVMContext &Ctx, EVT VT) const;
};
```

Each target sets up these actions in its `TargetLowering` constructor:

```cpp
X86TargetLowering::X86TargetLowering(const X86TargetMachine &TM,
                                      const X86Subtarget &STI) {
  // i64 integer operations: legal on 64-bit, expand on 32-bit
  if (Subtarget.is64Bit()) {
    setOperationAction(ISD::MUL, MVT::i64, Legal);
  } else {
    setOperationAction(ISD::MUL, MVT::i64, Expand);  // expand to 32-bit mul
    setOperationAction(ISD::UDIV, MVT::i64, LibCall); // use __udivdi3
  }

  // Vector operations:
  if (Subtarget.hasAVX2()) {
    setOperationAction(ISD::SMULO, MVT::v8i32, Legal);
  } else {
    setOperationAction(ISD::SMULO, MVT::v8i32, Expand);
  }
}
```

### 83.3.2 LowerOperation

For operations with the `Custom` action, `TargetLowering::LowerOperation` is called:

```cpp
SDValue X86TargetLowering::LowerOperation(SDValue Op,
                                           SelectionDAG &DAG) const {
  switch (Op.getOpcode()) {
  case ISD::VASTART:      return LowerVASTART(Op, DAG);
  case ISD::UREM:         return LowerUREM(Op, DAG);
  case ISD::ATOMIC_FENCE: return LowerATOMIC_FENCE(Op, DAG);
  case ISD::BUILD_VECTOR: return LowerBUILD_VECTOR(Op, DAG);
  // ...
  }
}
```

The `LowerOperation` function converts the high-level ISD node to a sequence of ISD nodes that the target can select.

### 83.3.3 Calling Convention Lowering

`TargetLowering` provides `LowerFormalArguments`, `LowerCall`, and `LowerReturn` hooks for lowering function entries, call sites, and return instructions:

```cpp
// Lowering a call:
SDValue X86TargetLowering::LowerCall(TargetLowering::CallLoweringInfo &CLI,
                                      SmallVectorImpl<SDValue> &InVals) const {
  // Assign args to registers/stack using CC_X86_64_C (from TableGen):
  CCState CCInfo(CLI.CallConv, CLI.IsVarArg, MF, RVLocs, *DAG.getContext());
  CCInfo.AnalyzeCallOperands(Outs, CC_X86_64_C);

  // Emit argument copies:
  for (auto &ArgLoc : CCInfo.getLocs()) {
    if (ArgLoc.isRegLoc())
      DAG.getCopyToReg(Chain, DL, ArgLoc.getLocReg(), ArgVal);
    else
      DAG.getStore(...); // store to stack
  }

  // Emit the call instruction:
  return DAG.getNode(X86ISD::CALL, DL, ...);
}
```

### 83.3.4 FastISel

`FastISel` (in `llvm/CodeGen/FastISel.h`) is a simplified, fast instruction selection path used at `-O0`. Each target implements a `FastISel` subclass that handles the common cases:

```cpp
bool X86FastISel::X86SelectCall(const Instruction *I) {
  // Fast path: handle the most common call patterns without full lowering
  if (CallInst *CI = dyn_cast<CallInst>(I)) {
    if (auto *Fn = CI->getCalledFunction()) {
      if (isIntrinsicCall(Fn)) return X86SelectIntrinsic(I);
      return X86SelectCallLowering(CI);
    }
  }
  return false; // fall back to SelectionDAG
}
```

FastISel is activated at `-O0` to minimize compile time; it produces suboptimal code but very fast. At `-O1` and above, SelectionDAG or GlobalISel handles instruction selection.

## 83.4 Register Classes

### 83.4.1 RegisterClass Definition

A **register class** is a set of registers that can hold values of a specific type, along with allocation metadata:

```tablegen
// AArch64 general-purpose registers:
def GPR64 : RegisterClass<"AArch64", [i64], 64, (add
  X0, X1, X2, X3, X4, X5, X6, X7,    // argument registers (also return)
  X8,                                   // indirect result location
  X9, X10, X11, X12, X13, X14, X15,  // caller-saved temporaries
  X16, X17,                            // IP0, IP1 (inter-procedure scratch)
  X18,                                 // platform register (ShadowCallStack)
  X19, X20, X21, X22, X23, X24, X25, X26, X27, X28,  // callee-saved
  FP, LR, SP, XZR                     // special purpose
)> {
  let CoveredBySubRegs = 1;
  let AllocationPriority = 2;
}
```

The `(add ...)` list defines both the members and their allocation order (the first elements are allocated first). The register allocator respects this order as a preference.

### 83.4.2 Register Banks

**Register banks** (GlobalISel) are coarser-grained groupings that span multiple register classes:

```cpp
// AArch64 register banks:
// GPRRegBank: GPR32, GPR64, WZR, XZR
// FPRRegBank: FPR8, FPR16, FPR32, FPR64, FPR128, FPR64_x

const RegisterBank *AArch64RegisterBankInfo::getRegBankForRegClass(
    const TargetRegisterClass &RC, const DataLayout &DL) const {
  if (AArch64::GPR64RegClass.hasSubClass(&RC))
    return &AArch64::GPRRegBank;
  if (AArch64::FPR64RegClass.hasSubClass(&RC))
    return &AArch64::FPRRegBank;
  // ...
}
```

The `RegBankSelect` pass assigns a register bank to each virtual register before the final instruction selection step.

## 83.5 SchedMachineModel and Instruction Itineraries

### 83.5.1 The Scheduling Model

The `MCSchedModel` (generated by `-gen-sched-models`) describes a CPU's microarchitecture for instruction scheduling:

```tablegen
def SkylakeClientModel : SchedMachineModel {
  let IssueWidth = 4;           // 4-wide superscalar
  let MicroOpBufferSize = 224;  // reorder buffer size
  let LoadLatency = 4;          // L1 cache hit latency
  let MispredictPenalty = 14;   // branch misprediction cycles
  let CompleteModel = 1;
}

// Resources (functional units):
def SKLPort0 : ProcResource<1>;   // ALU, FP, branch
def SKLPort1 : ProcResource<1>;   // ALU, FP
def SKLPort5 : ProcResource<1>;   // vector ALU
def SKLPort0156 : ProcResGroup<[SKLPort0, SKLPort1, SKLPort5, SKLPort6]>;

// Instruction → resource usage:
def : WriteRes<WriteIALU, [SKLPort0156]> {
  let Latency = 1;
  let NumMicroOps = 1;
}
def : WriteRes<WriteIMul32, [SKLPort1]> {
  let Latency = 3;
  let NumMicroOps = 1;
}
```

The scheduling model is used by both pre-RA scheduling (`MachineScheduler`) and post-RA scheduling (`PostRAScheduler`) to avoid resource conflicts and minimize critical-path latency.

### 83.5.2 Instruction Itineraries (Legacy)

Older targets (MIPS, ARM older) use **instruction itineraries** — a per-stage breakdown of instruction execution:

```tablegen
def IIC_iALUi : InstrItinClass;  // integer ALU with immediate

def CortexA9Itineraries : ProcessorItineraries<[A9_P0, A9_P1, A9_Memory], [], [
  InstrItinData<IIC_iALUi, [InstrStage<1, [A9_P0]>], [2, 1]>
  // 1 cycle on port P0; result latency 2 cycles, read-at-stage 1 cycle
]>;
```

Itineraries are less expressive than the `SchedMachineModel` approach; new targets should use `SchedMachineModel`.

## 83.6 Predicates and Features

### 83.6.1 SubtargetFeature

`SubtargetFeature` (in `llvm/MC/SubtargetFeature.h`) represents a CPU feature that can be enabled/disabled:

```tablegen
def FeatureSVE : SubtargetFeature<"sve", "HasSVE", "false",
    "Enable Scalable Vector Extension (SVE) instructions">;

def FeatureSVE2 : SubtargetFeature<"sve2", "HasSVE2", "false",
    "Enable Scalable Vector Extension 2 (SVE2) instructions",
    [FeatureSVE]>;
```

The dependencies list ensures that enabling SVE2 also enables SVE.

### 83.6.2 Function-Level Feature Override

```cpp
// __attribute__((target("sve2"))) on a function:
// → Clang adds "target-features" = "+sve2,+sve" to function attributes
// → TargetMachine creates a separate AArch64Subtarget with SVE2 enabled
// → That subtarget is used for instruction selection within that function
```

This mechanism enables function multiversioning (multiple implementations of the same function, selected at runtime by IFUNC) and architecture-specific optimization of hot functions.

---

## Chapter Summary

- **`TargetMachine`** is the root of the target description; it holds the target triple, data layout, code and relocation models, and creates the pass configuration. It provides per-function subtargets keyed on CPU + feature strings.
- **`TargetSubtargetInfo`** holds CPU-specific configuration: enabled features (generated by `-gen-subtarget`), scheduling model, and cache geometry. The subtarget is the key parameter for TTI cost queries.
- **`TargetLowering`** controls how IR operations map to SelectionDAG: `Legal` (directly supported), `Custom` (call `LowerOperation`), `Expand` (split into simpler ops), `Promote` (widen type), or `LibCall` (runtime function). It also provides `LowerCall`/`LowerReturn`/`LowerFormalArguments` for calling convention lowering.
- **Register classes** (from `-gen-register-info`) group physical registers by type; their allocation order guides the register allocator. **Register banks** (GlobalISel) are coarser groupings across classes.
- **`SchedMachineModel`** describes microarchitecture (issue width, reorder buffer, functional units) and maps instructions to resource usage and latencies; it drives both pre-RA and post-RA scheduling.
- **`SubtargetFeature`** represents CPU capabilities with dependency chains; per-function `__attribute__((target(...)))` creates isolated subtargets with different feature sets, enabling function multiversioning.
