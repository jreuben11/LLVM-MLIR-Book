# Chapter 117 — DWARF and Debug Info

*Part XVI — JIT, Sanitizers, and Diagnostic Tools*

DWARF (Debugging With Attributed Record Formats) is the dominant debug information format on Linux and macOS. Generated by every Clang, GCC, and Rust compilation with debug flags enabled, DWARF encodes the relationship between machine instructions and source-level constructs: functions, variables, types, source lines, and call frames. This information enables debuggers to display stack traces, evaluate expressions, step through source lines, display variable values, and unwind call stacks for both debugging and C++ exception handling. This chapter examines the DWARF format in depth — its section layout, DIE hierarchy, location expressions, line tables, and call frame information — along with LLVM's toolchain for generating and inspecting DWARF, and advanced features like split DWARF for reduced link times.

---

## 117.1 DWARF Overview

### The Standard

DWARF is an open standard maintained by the DWARF standards committee. Current version: DWARF 5 (2017). DWARF 4 (2010) is still widely deployed; DWARF 5 adds material improvements for performance and modern language features.

| Version | Year | Key Additions |
|---------|------|---------------|
| DWARF 2 | 1993 | Original standardization; `.debug_info`, `.debug_line`, `.debug_frame` |
| DWARF 3 | 2005 | Namespace support, complex location expressions, `.debug_types` |
| DWARF 4 | 2010 | Type units, `.debug_types`, packed encoding improvements |
| DWARF 5 | 2017 | `.debug_str_offsets`, `.debug_rnglists`, `.debug_loclists`, `.debug_addr`, line table v5, index sections |

### ELF Sections

DWARF data is stored in dedicated ELF sections:

| Section | Content |
|---------|---------|
| `.debug_info` | Core debugging data — DIE tree for each compilation unit |
| `.debug_abbrev` | Abbreviation table — compact encoding of DIE tag + attribute combinations |
| `.debug_str` | String table — pooled strings referenced by `.debug_info` |
| `.debug_str_offsets` | (DWARF 5) Per-CU offsets into `.debug_str` |
| `.debug_line` | Line number tables — address → source file/line mappings |
| `.debug_loc` | (DWARF 4) Location list entries |
| `.debug_loclists` | (DWARF 5) Replacement for `.debug_loc` with better encoding |
| `.debug_ranges` | (DWARF 4) Address ranges for non-contiguous functions |
| `.debug_rnglists` | (DWARF 5) Replacement for `.debug_ranges` |
| `.debug_aranges` | Address ranges → compilation unit (fast lookup index) |
| `.debug_frame` | Call frame information (CFA and register save locations) |
| `.eh_frame` | Exception handling frame info (for C++ exceptions and libunwind) |
| `.debug_pubnames` | (DWARF 4) Function/variable name lookup (superseded by `.debug_names` in DWARF 5) |
| `.debug_names` | (DWARF 5) Accelerator table for fast name-to-DIE lookup |
| `.debug_addr` | (DWARF 5) Address table referenced by DW_FORM_addrx |
| `.debug_types` | (DWARF 4) Type unit sections (rarely used in practice) |

### macOS dSYM Bundles

macOS debuggers (LLDB, Instruments) use `.dSYM` bundles — directories containing the DWARF debug info extracted from the binary:

```
MyApp.app.dSYM/
└── Contents/
    ├── Info.plist
    └── Resources/
        └── DWARF/
            └── MyApp      # ELF/Mach-O with only debug sections
```

`dsymutil` (part of Apple's toolchain) links debug info from object files into a dSYM bundle, stripping it from the main binary. LLDB looks up dSYM bundles via Spotlight or explicit paths.

## 117.2 DIE Structure

### Compilation Units

The `.debug_info` section is organized as a sequence of **Compilation Units (CUs)**. Each CU corresponds to one translation unit (one `.cpp` file). A CU starts with a header:

```
CU Header:
  unit_length: u32 (total length excluding this field)
  version: u16 (DWARF version, e.g. 5)
  unit_type: u8 (DW_UT_compile = 0x01)
  address_size: u8 (4 or 8 bytes)
  debug_abbrev_offset: u32 (offset into .debug_abbrev)
```

Following the header is a tree of **DIEs** (Debug Information Entries). Each DIE has:
- A **tag** (`DW_TAG_*`): the kind of entity (function, variable, type, etc.)
- A list of **attributes** (`DW_AT_*`): properties of the entity (name, type, address, etc.)
- Optional **children** (indicated by `DW_CHILDREN_yes` in the abbreviation entry)

### Common DIE Tags

| Tag | Meaning |
|-----|---------|
| `DW_TAG_compile_unit` | Root of a compilation unit |
| `DW_TAG_subprogram` | A function or method |
| `DW_TAG_variable` | A variable (local or global) |
| `DW_TAG_formal_parameter` | A function parameter |
| `DW_TAG_base_type` | A primitive type (`int`, `float`, etc.) |
| `DW_TAG_pointer_type` | A pointer type |
| `DW_TAG_structure_type` | A struct or class |
| `DW_TAG_union_type` | A union type |
| `DW_TAG_array_type` | An array type |
| `DW_TAG_typedef` | A typedef |
| `DW_TAG_namespace` | A C++ namespace |
| `DW_TAG_class_type` | A C++ class |
| `DW_TAG_inheritance` | A base class relationship |
| `DW_TAG_member` | A struct/class member |
| `DW_TAG_subrange_type` | Array bounds |
| `DW_TAG_inlined_subroutine` | An inlined function call site |
| `DW_TAG_lexical_block` | A lexical scope (C block) |
| `DW_TAG_template_type_parameter` | C++ template type argument |

### An Annotated Example

Consider this C++ function:

```cpp
int square(int x) {
    return x * x;
}
```

The DWARF DIE tree for this function:

```
DW_TAG_subprogram
  DW_AT_name: "square"
  DW_AT_low_pc: 0x401000           (start address)
  DW_AT_high_pc: 0x401010          (length in bytes, DWARF 4+)
  DW_AT_type: → DW_TAG_base_type "int"
  DW_AT_decl_file: 1               (index into file table)
  DW_AT_decl_line: 1
  DW_AT_external: DW_FORM_flag 1   (globally visible)
  DW_AT_frame_base: DW_OP_call_frame_cfa
  
  DW_TAG_formal_parameter
    DW_AT_name: "x"
    DW_AT_type: → DW_TAG_base_type "int"
    DW_AT_location: DW_OP_reg5     (in %rdi on entry)
    DW_AT_decl_line: 1
```

`llvm-dwarfdump` output for this:

```bash
llvm-dwarfdump --debug-info square.o
# 0x00000024: DW_TAG_subprogram
#               DW_AT_low_pc	(0x0000000000000000)
#               DW_AT_high_pc	(0x0000000000000010)
#               DW_AT_frame_base	(DW_OP_call_frame_cfa)
#               DW_AT_name	("square")
#               DW_AT_decl_file	("/tmp/square.cpp")
#               DW_AT_decl_line	(1)
#               DW_AT_type	(0x00000050 "int")
#               DW_AT_external	(true)
# 0x0000003d:   DW_TAG_formal_parameter
#                 DW_AT_name	("x")
#                 DW_AT_type	(0x00000050 "int")
#                 DW_AT_location	(DW_OP_reg5 RDI)
```

### The Abbreviation Table

Raw DWARF encoding would be enormous if every DIE listed all its tag/attribute pairs explicitly. The abbreviation table (`.debug_abbrev`) provides a compact coding:

```
// .debug_abbrev entry for DW_TAG_subprogram:
abbrev_code: 3
  tag: DW_TAG_subprogram
  children: DW_CHILDREN_yes
  attributes:
    DW_AT_name,       DW_FORM_strp    (4-byte offset into .debug_str)
    DW_AT_low_pc,     DW_FORM_addr    (address-width bytes)
    DW_AT_high_pc,    DW_FORM_udata   (ULEB128 length)
    DW_AT_type,       DW_FORM_ref4    (4-byte offset into CU)
    DW_AT_external,   DW_FORM_flag_present  (implicit true)
    0, 0  (terminator)
```

A `.debug_info` DIE then just stores `abbrev_code=3` followed by the attribute values in the specified forms — without repeating the tag or attribute codes. This compression is significant: typical DWARF is 3–5× the size of the code, but would be 10× without abbreviations.

### DWARF Forms

Attributes have types (forms) that specify how the value is encoded:

| Form | Encoding |
|------|---------|
| `DW_FORM_addr` | Architecture-width address |
| `DW_FORM_string` | Null-terminated inline string |
| `DW_FORM_strp` | 4-byte offset into `.debug_str` |
| `DW_FORM_strx` | (DWARF 5) ULEB128 index into `.debug_str_offsets` |
| `DW_FORM_data1/2/4/8` | Fixed-width integer |
| `DW_FORM_udata/sdata` | ULEB128/SLEB128 variable-length integer |
| `DW_FORM_ref4` | 4-byte offset relative to CU start |
| `DW_FORM_ref_addr` | Global offset into `.debug_info` |
| `DW_FORM_exprloc` | ULEB128 length + location expression bytes |
| `DW_FORM_flag` | 1-byte boolean |
| `DW_FORM_flag_present` | Implicit `true`, no bytes stored |
| `DW_FORM_addrx` | (DWARF 5) Index into `.debug_addr` |

## 117.3 Location Expressions

### Purpose

A DWARF location expression answers: "At this point in program execution, where does this variable live?" Variables may live in registers, on the stack, in heap memory, be constant-folded away, or move between locations as optimizations change register allocation across the function body.

### The Location Expression Machine

Location expressions are byte sequences that drive a stack-based virtual machine. The machine has:
- A value stack of (type, value) pairs
- Access to the current register set
- Access to process memory
- The CFA (Canonical Frame Address)

Common operations:

```
DW_OP_lit5          — push literal 5
DW_OP_reg5          — push value of register 5 (rdi on x86-64)
DW_OP_breg5 offset  — push (reg5 + SLEB128 offset)
DW_OP_fbreg offset  — push (frame_base + SLEB128 offset)
DW_OP_addr addr     — push compile-time address
DW_OP_deref         — pop address, push *address
DW_OP_deref_size N  — pop address, push N-byte value at *address
DW_OP_plus          — pop two values, push sum
DW_OP_plus_uconst N — pop, add ULEB128 constant N
DW_OP_and           — bitwise AND
DW_OP_shr           — shift right
DW_OP_piece N       — N bytes of object from top of stack
DW_OP_stack_value   — value (not address) is on stack
DW_OP_entry_value   — value of expression at function entry
DW_OP_call_frame_cfa — push CFA (canonical frame address)
```

### Examples

**Variable in register (ABI argument passing)**:
```
DW_AT_location: DW_OP_reg0 (RAX)
// Variable x lives in %rax at this point
```

**Variable on stack**:
```
DW_AT_location: DW_OP_fbreg -24
// Variable x lives at (frame_base - 24), typically (rbp - 24)
```

**Pointer dereference**:
```
DW_AT_location: DW_OP_breg6 8, DW_OP_deref
// Variable x lives at *(rbp + 8)
// (e.g., x is at the memory location pointed to by rbp+8)
```

**Split value (struct field split between register and memory)**:
```
DW_AT_location:
  DW_OP_reg0; DW_OP_piece 4   // low 4 bytes from %rax
  DW_OP_reg1; DW_OP_piece 4   // high 4 bytes from %rdx
```

This last form handles the x86-64 ABI's return of `struct { int a, b; }` split across `%rax`/`%rdx`.

### Location Lists

When a variable moves between locations across the function body (common after optimization), a single location expression is insufficient. Location lists map PC ranges to location expressions:

```
// .debug_loclists entry for variable 'x':
[0x401000, 0x401010): DW_OP_reg0     // x lives in %rdi (entry to first store)
[0x401010, 0x401020): DW_OP_fbreg -8 // x was spilled to stack
[0x401020, 0x401040): DW_OP_reg3     // x was reloaded into %rbx
```

In DWARF 5, location lists are stored in `.debug_loclists` with a more compact encoding (delta-encoded addresses relative to the CU's base address).

## 117.4 Line Tables

### Purpose

The line number table maps ranges of machine instructions to source file/line/column tuples. Debuggers use this to:
- Translate addresses to source locations (for stack traces)
- Translate source locations to addresses (for `break foo.cpp:42`)
- Step "to next source line" by computing the next line's address range

### The State Machine

DWARF encodes line tables as a sequence of opcodes that drive a virtual state machine. The machine state:

```
address        : current machine instruction address
op_index       : for VLIW architectures; normally 0
file           : index into the file table
line           : source line number (starts at 1)
column         : source column number (starts at 1)
is_stmt        : this address is a recommended breakpoint location
basic_block    : this is the start of a basic block
end_sequence   : this address is the end of a sequence
prologue_end   : this is the end of a function prologue (GDB sets breakpoint here)
epilogue_begin : this is the beginning of a function epilogue
```

Operations include:
- **Special opcodes**: compact encoding of common (address delta, line delta) combinations
- **Standard opcodes**: `DW_LNS_copy` (emit a row), `DW_LNS_advance_pc`, `DW_LNS_advance_line`, `DW_LNS_set_file`, `DW_LNS_set_column`, `DW_LNS_negate_stmt`, `DW_LNS_set_prologue_end`
- **Extended opcodes**: `DW_LNE_set_address` (absolute address), `DW_LNE_end_sequence`

### DWARF 5 Line Table Improvements

DWARF 5 replaced the directory/filename encoding with `DW_LNCT_*` content type descriptors, enabling:
- MD5 checksums for source files (allow debuggers to warn when source doesn't match binary)
- `DW_LNCT_source` (inline source text)
- More compact path encoding via `DW_FORM_line_strp`

```bash
# View line table for a binary:
llvm-dwarfdump --debug-line /usr/lib/llvm-22/bin/clang-22 2>/dev/null | head -80

# Output:
# .debug_line contents:
# debug_line[0x000000]
# Line table prologue:
#   total_length: 0x00001234
#   version: 5
#   address_size: 8
#   seg_select_size: 0
#   ...
#   file_names[  0]:
#              name: "CodeGenerator.cpp"
#           dir_idx: 5
#               md5: (0x7f3a9b2ec3f840d9a3de0a17b8a2e49f)
# Address            Line   Column File   ISA Discriminator OpIndex Flags
# ------------------ ------ ------ ------ --- ------------- ------- -------------
# 0x0000000000401000      1      0      1   0             0       0  is_stmt
# 0x0000000000401008      2      4      1   0             0       0  is_stmt prologue_end
```

## 117.5 Call Frame Information (CFI)

### Purpose

CFI (`.debug_frame` / `.eh_frame`) enables debuggers to unwind stack frames: given the current instruction pointer, recover the stack pointer, frame pointer, and saved registers of the caller. This makes `bt` (backtrace) work in debuggers and enables C++ exception handling (libunwind uses `.eh_frame` to find the catch block for a thrown exception).

### CIE and FDE

CFI is encoded as a sequence of CIEs (Common Information Entries) and FDEs (Frame Description Entries):

**CIE** (one per function type or code alignment):
```
CIE:
  length: u32
  CIE_id: 0xFFFFFFFF (for .debug_frame) or 0 (for .eh_frame)
  version: u8
  augmentation: string (e.g., "zR" for FDE encoding info)
  code_alignment_factor: ULEB128 (e.g., 1 for x86, 4 for ARM)
  data_alignment_factor: SLEB128 (e.g., -8 for x86-64)
  return_address_register: ULEB128 (e.g., 16 = RIP on x86-64)
  initial_instructions: CFA instructions (apply to all FDEs referencing this CIE)
```

**FDE** (one per function):
```
FDE:
  length: u32
  CIE_pointer: u32 (offset back to CIE)
  initial_location: addr (function start PC)
  address_range: addr (function length)
  instructions: CFA instructions (specific to this function)
```

### CFA Instructions

CFA instructions describe how to compute the **Canonical Frame Address** (CFA) — typically the value of the stack pointer at the call instruction in the caller — and where saved registers are located.

| Instruction | Effect |
|-------------|--------|
| `DW_CFA_def_cfa reg, offset` | CFA = reg + offset |
| `DW_CFA_def_cfa_register reg` | CFA = reg + current_offset |
| `DW_CFA_def_cfa_offset offset` | CFA = current_reg + offset |
| `DW_CFA_offset reg, factor` | reg saved at CFA + factor * data_alignment |
| `DW_CFA_register reg1, reg2` | reg1 saved in reg2 |
| `DW_CFA_restore reg` | restore reg to initial rule |
| `DW_CFA_advance_loc delta` | advance PC by delta * code_alignment |
| `DW_CFA_remember_state` | push current CFA rules to stack |
| `DW_CFA_restore_state` | pop CFA rules from stack |

### Example: x86-64 Function Prologue

A standard x86-64 function prologue and its corresponding CFI:

```asm
; Function start (CFA = rsp + 8, since CALL pushed return address)
foo:
    .cfi_startproc
    .cfi_def_cfa rsp, 8         ; CFA = rsp + 8

    push rbp                     ; rsp -= 8; [rsp] = rbp
    .cfi_def_cfa_offset 16      ; CFA = rsp + 16 (rsp moved by -8)
    .cfi_offset rbp, -16        ; rbp saved at CFA - 16 = [rsp]

    mov  rbp, rsp
    .cfi_def_cfa_register rbp   ; CFA = rbp + 16 (CFA now rbp-relative)

    ; ... function body ...

    pop  rbp
    .cfi_def_cfa rsp, 8         ; restore: CFA = rsp + 8
    ret
    .cfi_endproc
```

Given this CFI and a stopped program at any instruction in `foo`, a debugger can:
1. Compute CFA = rbp + 16 (from `DW_CFA_def_cfa_register rbp`)
2. Find return address at CFA - 8 (standard: return address is at [rsp] before the call, i.e., CFA - 8)
3. Find caller's rbp at CFA - 16 (from `DW_CFA_offset rbp, -16`)
4. Reconstruct the caller's frame

### `.eh_frame` vs `.debug_frame`

`.eh_frame` and `.debug_frame` contain similar information but serve different purposes:

- `.debug_frame`: consumed by debuggers; may not be present in stripped binaries
- `.eh_frame`: consumed by `libunwind`/`libgcc_s` for C++ exception handling; must be present in any binary that uses C++ exceptions

Both use the same CFA instruction encoding. The main difference is the CIE `CIE_id` marker and augmentation string format.

## 117.6 LLVM's Debug Info Generation

### The DIBuilder API

LLVM provides a C++ API for constructing DWARF debug metadata: [`llvm/IR/DIBuilder.h`](https://github.com/llvm/llvm-project/blob/llvmorg-22.1.0/llvm/include/llvm/IR/DIBuilder.h).

```cpp
#include "llvm/IR/DIBuilder.h"
#include "llvm/IR/Module.h"

// Setup:
llvm::Module M("my_module", Context);
llvm::DIBuilder DBuilder(M);

// Create compile unit:
llvm::DIFile *File = DBuilder.createFile("foo.cpp", "/path/to/");
llvm::DICompileUnit *CU = DBuilder.createCompileUnit(
    llvm::dwarf::DW_LANG_C_plus_plus_14, File,
    "clang version 22.1.3",   // producer
    /*isOptimized=*/false,
    /*Flags=*/"",
    /*RuntimeVersion=*/0);

// Create type for 'int':
llvm::DIBasicType *IntTy = DBuilder.createBasicType(
    "int", 32, llvm::dwarf::DW_ATE_signed);

// Create a function type:
llvm::DISubroutineType *FuncTy = DBuilder.createSubroutineType(
    DBuilder.getOrCreateTypeArray({IntTy, IntTy}));  // returns int, takes int

// Create function debug info:
llvm::DISubprogram *SP = DBuilder.createFunction(
    CU, "square", /*LinkageName=*/"_Z6squarei",
    File, /*LineNo=*/1,
    FuncTy,
    /*ScopeLine=*/1,
    llvm::DINode::FlagPrototyped,
    llvm::DISubprogram::SPFlagDefinition);
F->setSubprogram(SP);

// Create local variable:
llvm::DILocalVariable *VarX = DBuilder.createParameterVariable(
    SP, "x", /*ArgNo=*/1, File, /*LineNo=*/1, IntTy,
    /*alwaysPreserve=*/false);

// Attach debug location to instructions:
llvm::IRBuilder<> IRB(BasicBlock);
IRB.SetCurrentDebugLocation(
    llvm::DILocation::get(Context, /*Line=*/2, /*Col=*/12, SP));

// Emit DW_AT_location for the variable:
DBuilder.insertDeclare(AllocaForX, VarX,
                        DBuilder.createExpression(),
                        llvm::DILocation::get(Context, 1, 0, SP),
                        FirstInst);

// Finalize (must be called before emitting):
DBuilder.finalize();
```

### DwarfDebug: The Emission Driver

[`llvm/lib/CodeGen/AsmPrinter/DwarfDebug.cpp`](https://github.com/llvm/llvm-project/blob/llvmorg-22.1.0/llvm/lib/CodeGen/AsmPrinter/DwarfDebug.cpp) is the central class that drives DWARF emission during code generation. It:

1. Receives machine instruction annotations (debug location metadata on MachineInstrs)
2. Builds CUs, DIEs, abbreviation tables
3. Emits `.debug_info`, `.debug_abbrev`, `.debug_line`, `.debug_loc`/`.debug_loclists`
4. Handles location tracking across register allocation (variable location changes)
5. Emits `.cfi_*` directives for `.debug_frame`

Key hooks:
```cpp
// Called at the start of each function:
void DwarfDebug::beginFunction(const MachineFunction *MF);
// Called after each instruction:
void DwarfDebug::beginInstruction(const MachineInstr *MI);
// Called at the end of each function:
void DwarfDebug::endFunction(const MachineFunction *MF);
```

### Salvaging Debug Info

When an LLVM pass removes or transforms an instruction that carries debug location metadata, it should call `llvm::salvageDebugInfo()`:

```cpp
// When removing instruction I:
llvm::salvageDebugInfoForDbgValues(*I, DeadInsts);
// This attempts to rewrite debug uses of I's value
// in terms of I's operands (e.g., DW_OP_plus, DW_OP_deref)
// so the variable is still visible in the debugger after optimization
```

Without salvaging, optimization passes leave variables as "optimized out" even when the value could be reconstructed. The LLVM test suite (`check-llvm`) includes tests specifically for debug info preservation through optimization passes.

## 117.7 llvm-dwarfdump and Debug Tools

### llvm-dwarfdump

`llvm-dwarfdump` is LLVM's DWARF inspection tool, useful for debugging debug info itself:

```bash
# Dump all DWARF sections:
llvm-dwarfdump --all binary

# Dump only .debug_info:
llvm-dwarfdump --debug-info binary

# Verify DWARF integrity (checks cross-references, encoding):
llvm-dwarfdump --verify binary
# Outputs: DWARF verification results:
#   Number of .debug_info errors: 0
#   DWARF verification succeeded.

# Look up by address:
llvm-dwarfdump --lookup=0x401020 binary
# Output: source file, line, function, variable locations at that PC

# Show line table:
llvm-dwarfdump --debug-line binary | head -60

# Show call frame info:
llvm-dwarfdump --debug-frame binary | head -40

# Show location lists:
llvm-dwarfdump --debug-loclists binary

# Verbose: show DIE tree with offsets:
llvm-dwarfdump --debug-info --verbose binary | head -100
```

The `--verify` option is particularly useful for catching LLVM codegen bugs that produce malformed DWARF — a common class of bugs found by downstream tools.

### addr2line

`addr2line` (from GNU binutils) and `llvm-addr2line` (LLVM's compatible implementation) translate addresses to source locations:

```bash
# Translate an address:
llvm-addr2line -e binary -f 0x401020
# Output:
# square
# /path/to/source.cpp:3

# With inlined functions (-i):
llvm-addr2line -e binary -f -i 0x401020
# Output (inlined call chain):
# inner_func
# /path/to/source.cpp:10
# outer_func
# /path/to/source.cpp:20
# main
# /path/to/source.cpp:30
```

### llvm-symbolizer

`llvm-symbolizer` is more powerful than `addr2line`:

```bash
# Symbolize an address:
echo "0x401020" | llvm-symbolizer --exe=binary

# With inlining:
echo "0x401020" | llvm-symbolizer --exe=binary --inlining=true

# JSON output (for programmatic consumption):
echo "0x401020" | llvm-symbolizer --exe=binary --output-style=JSON

# Symbolize from a crash report:
# Extract addresses from a stack trace and pipe through llvm-symbolizer
```

Sanitizer runtimes (ASan, MSan, etc.) use `llvm-symbolizer` internally to translate stack frame addresses to source locations in their reports.

### pahole

[`pahole`](https://git.kernel.org/pub/scm/devel/pahole/pahole.git) analyzes struct layouts from DWARF:

```bash
# Show struct layout with holes:
pahole --class_name=MyStruct binary
# Output:
# struct MyStruct {
#         int                        a;                    /*     0     4 */
#         /* XXX 4 bytes hole, try to pack */
#         double                     b;                    /*     8     8 */
#         char                       c;                    /*    16     1 */
#         /* XXX 7 bytes padding */
#
#         /* size: 24, cachelines: 1, members: 3 */
#         /* sum members: 13, holes: 1, sum holes: 4 */
#         /* padding: 7 */
#         /* last cacheline: 24 bytes */
# };

# Find structs with padding:
pahole --show_holes binary | grep -A5 "holes:"
```

`pahole` is also used to generate BTF (BPF Type Format) from kernel DWARF for BPF programs.

## 117.8 Split DWARF and DWP

### The Link-Time DWARF Problem

In large C++ projects, DWARF can be 5–10× the size of the code. Linking a large binary (e.g., a game engine or Chromium) requires the linker to process gigabytes of DWARF from hundreds of object files. This is slow and consumes enormous memory. Two solutions exist: split DWARF (move debug info to separate files) and indexed DWARF (`.debug_names` for fast lookup without reading all DIEs).

### Split DWARF: The Skeleton Approach

`-gsplit-dwarf` (Clang/GCC flag) splits debug info:

1. **Skeleton CU**: remains in the `.o` file. Contains minimal info (producer, path, DW-unit-ID) to identify the full CU. Has `DW_AT_dwo_name` pointing to the `.dwo` file.

2. **Full DWARF object (`.dwo` file)**: contains all DIEs, abbreviation table, string table, location lists, etc. Not included in the linked binary.

```bash
# Compile with split DWARF:
clang-22 -gsplit-dwarf -g -c source.cpp -o source.o
ls
# source.o   (small; skeleton CU + relocatable code)
# source.dwo (large; full DWARF)

# Link: .dwo files are NOT included:
clang-22 source.o -o binary
# binary contains skeleton CUs; debugger must find .dwo files

# Package .dwo files into a single .dwp file:
llvm-dwp source.dwo other.dwo -o binary.dwp
# Debugger looks for binary.dwp alongside binary
```

### Skeleton vs Full CU

The skeleton CU in the `.o` file contains:

```
DW_TAG_compile_unit
  DW_AT_dwo_name: "source.dwo"
  DW_AT_dwo_id: 0x7f3a9b2ec3f840d9  (64-bit hash linking skeleton to .dwo)
  DW_AT_low_pc: 0x0
  DW_AT_ranges: ...  (address ranges for this CU)
  DW_AT_stmt_list: offset  (line table, kept in .o)
  DW_AT_comp_dir: "/path/to/src"
```

The `.dwo` file's CU has `DW_AT_dwo_id` matching the skeleton's, plus all the full DIE content.

### DW_AT_dwo_id and Staleness Detection

The DWO ID is a hash of the compile unit's content. If the source is recompiled, the DWO ID changes and the old `.dwo` file is stale. Debuggers check the DWO IDs match before using a `.dwo` file.

### DWP Format

A DWARF Package (`.dwp`) merges multiple `.dwo` files for deployment convenience:

```bash
# Merge all .dwo files for a binary:
llvm-dwp $(find . -name "*.dwo") -o myapp.dwp

# Debug: LLDB looks for myapp.dwp in same directory as myapp
(lldb) target create ./myapp
# LLDB automatically finds myapp.dwp
```

The `.dwp` format uses DWARF 5's index sections (`.debug_cu_index`, `.debug_tu_index`) for fast per-function lookup by DWO ID without scanning all packages.

### Index Sections for Fast Lookup

DWARF 5 `.debug_names` provides an accelerator table for name-to-DIE lookups:

```
.debug_names:
  Header (unit count, bucket count, etc.)
  CU offsets list
  Local type units list
  Foreign type units list
  Buckets (hash table)
  Hashes
  String offsets (into .debug_str)
  Entry pool (DIE tag + attribute coding + offset)
```

Without `.debug_names`, finding a function by name requires scanning all CUs — O(N) in the number of compilation units. With `.debug_names`, it's O(1) hash lookup. LLDB uses `.debug_names` (or its Apple-specific predecessor `.apple_names`) for fast symbol resolution.

```bash
# Check for .debug_names section:
llvm-dwarfdump --debug-names binary | head -20

# Generate .debug_names during link:
# LLD: --debug-names (default in LLVM 18+)
# ld.lld --debug-names myapp

# Or post-link with llvm-dwarfdump (for older setups):
# Not directly supported; regenerate with dsymutil on macOS
```

## Chapter Summary

- **DWARF 5** is the current standard, stored in `.debug_info`, `.debug_line`, `.debug_loclists`, `.debug_rnglists`, `.debug_names`, and related sections in ELF binaries.

- **DIEs** (Debug Information Entries) form a tree rooted at `DW_TAG_compile_unit`, encoding functions (`DW_TAG_subprogram`), variables (`DW_TAG_variable`), and types (`DW_TAG_base_type`, `DW_TAG_structure_type`). The abbreviation table compresses the encoding by factoring out repeated tag/attribute combinations.

- **Location expressions** are DWARF stack-machine programs that compute where a variable lives at runtime — in a register, on the stack, at a computed address, or in pieces across multiple locations. Location lists handle variables that move as the compiler reschedules code.

- **Line tables** are encoded as state machine programs that produce (address, file, line, column) rows. DWARF 5 adds MD5 checksums for source-to-binary correspondence verification.

- **Call Frame Information** (`.eh_frame` / `.debug_frame`) encodes the CFA computation and register save locations using CIEs and FDEs. This enables both debugger stack unwinding and C++ exception handling via `libunwind`.

- **`DIBuilder`** is LLVM's C++ API for constructing debug metadata in IR. `DwarfDebug` drives DWARF emission from machine IR annotations. `salvageDebugInfo()` preserves variable locations through optimization passes.

- **`llvm-dwarfdump`** inspects and verifies DWARF sections. `llvm-symbolizer`/`addr2line` translate addresses to source locations. `pahole` analyzes struct layouts.

- **Split DWARF** (`-gsplit-dwarf`) separates full debug info into `.dwo` files excluded from the linked binary, dramatically reducing link time. `llvm-dwp` merges `.dwo` files into `.dwp` packages. `.debug_names` provides O(1) name-to-DIE lookup.
