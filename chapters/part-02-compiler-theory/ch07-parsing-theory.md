# Chapter 7 — Parsing Theory

*Part II — Compiler Theory and Foundations*

Chapter 6 established that lexical analysis is a solved problem: regular languages, deterministic finite automata, and the linear-time scanning model provide a complete and practical foundation. Parsing is harder in every dimension — theoretically, algorithmally, and practically. The language of correctly-nested expressions is not regular; the language of type-correct C++ templates is not even context-free. The central achievement of formal language theory for compilers is the identification of the *context-free grammar* as the right formalism for the syntactic level, together with a family of efficient algorithms for deciding membership, constructing parse trees, and recovering from errors.

This chapter develops the complete theory of parsing, from the formal definition of context-free grammars through the practical decision of why production compilers write their parsers by hand. A single running grammar — the classical arithmetic expression grammar from [Dragon §4.1] — threads through the worked examples for LL(1) table construction, LR(0) automaton construction, LALR table derivation, and Earley parsing. By the end of this chapter you will understand why C++ parsing is undecidable in the general case, why LALR(1) is the sweet spot for generated parsers, why Pratt's algorithm handles expressions more cleanly than either LL or LR theory predicts, and why Clang chose a hand-written recursive-descent parser anyway.

Chapter 8 takes up where this chapter ends, covering semantic analysis: name resolution, type checking, and the scope rules that lie beyond the context-free boundary.

## 7.1 Context-Free Grammars: Formal Definition and Ambiguity

### Formal definition

A **context-free grammar** (CFG) is a 4-tuple G = (V, T, P, S) where [Dragon §4.2, EaC §3.1]:

- **V** is a finite set of **nonterminals** (also called *variables* or *syntactic categories*). By convention, written in uppercase or angle-brackets.
- **T** is a finite set of **terminals** (the token types produced by the lexer). T and V are disjoint.
- **P** is a finite set of **productions**, each of the form A → α where A ∈ V and α ∈ (V ∪ T)*. The right-hand side α is a (possibly empty) string of terminals and nonterminals.
- **S ∈ V** is the **start symbol**.

The **running grammar** used throughout this chapter is the left-recursive arithmetic expression grammar [Dragon §4.1]:

```
G₀:
    E  →  E + T  |  T
    T  →  T * F  |  F
    F  →  ( E )  |  id
```

Here V = {E, T, F}, T = {+, *, (, ), id}, S = E. This grammar encodes standard arithmetic precedence (multiplication binds tighter than addition) and left-associativity (both operators) through the structure of its productions.

### Derivations and the language L(G)

A **sentential form** is any string in (V ∪ T)* derivable from S by zero or more production applications. The **language** generated by G is:

```
L(G) = { w ∈ T* : S ⟹* w }
```

where ⟹* denotes the reflexive-transitive closure of the single-step derivation relation ⟹. A single step A ⟹ β[A→γ] replaces one occurrence of a nonterminal A with the right-hand side γ of some production A → γ, leaving the rest of the sentential form unchanged.

Two canonical orderings of derivation steps matter for parsing:

**Leftmost derivation (⟹_lm):** At each step, replace the *leftmost* nonterminal. The sequence of productions applied in a leftmost derivation corresponds exactly to a top-down left-to-right parse — the mental model for LL parsers.

**Rightmost derivation (⟹_rm):** At each step, replace the *rightmost* nonterminal. The reverse of a rightmost derivation — reading the sequence of applied productions backward — corresponds to a bottom-up right-to-left parse — the mental model for LR parsers.

**Parse tree:** The derivation structure of a string w ∈ L(G) can be represented as a rooted ordered tree where:
- The root is labeled S
- Each interior node is labeled by a nonterminal A
- The children of an interior node labeled A are labeled by the symbols of some production A → X₁X₂…Xₙ, left-to-right
- Each leaf is labeled by a terminal or ε

The **yield** of a parse tree is the left-to-right concatenation of its leaf labels. A parse tree for w ∈ L(G) is a tree with yield w. A leftmost derivation and a rightmost derivation of the same string *w* produce the same parse tree — the tree is independent of the order in which productions are applied, only of which productions are applied. This observation is fundamental: the parse tree is the semantic object we want; the derivation is just a way to construct it.

**Example:** One parse tree for `id + id * id` under G₀:

```
              E
           /  |  \
          E   +   T
          |      /|\
          T     T  *  F
          |     |     |
          F     F    id
          |     |
         id    id
```

The yield is `id + id * id`. The tree encodes that `*` binds more tightly than `+` because the second `id` and third `id` are combined into a T subtree before T is combined with E.

### Ambiguity

**Definition:** A CFG G is **ambiguous** if there exists some string w ∈ L(G) that has *two or more* distinct parse trees [Dragon §4.3.2, EaC §3.3]. Equivalently, w has two distinct leftmost derivations, or two distinct rightmost derivations.

**Ambiguity is a property of the grammar, not the language.** An *inherently ambiguous* language has no unambiguous grammar; the ambiguity is intrinsic to the language itself.

#### Classic example 1: Dangling else

The classic dangling-else grammar:

```
stmt  →  if expr then stmt
       |  if expr then stmt else stmt
       |  other
```

The string `if e₁ then if e₂ then s₁ else s₂` has two parse trees: one in which the `else` matches the outer `if` and one in which it matches the inner `if`. Both are valid derivations.

**Standard resolution:** Stratify the grammar into *matched* and *unmatched* statements [Dragon §4.3.2]:

```
stmt  →  matched_stmt  |  unmatched_stmt

matched_stmt    →  if expr then matched_stmt else matched_stmt
                |  other

unmatched_stmt  →  if expr then stmt
                |  if expr then matched_stmt else unmatched_stmt
```

This grammar is unambiguous: the `else` always matches the nearest preceding unmatched `if`. It is the standard textbook disambiguation, implemented by every C-family language.

In practice, parser generators like yacc/bison do not require the stratified grammar. Instead they declare a **shift-prefer rule**: when a shift-reduce conflict arises between shifting the `else` and reducing the inner statement, prefer the shift. This implicitly implements nearest-match semantics without rewriting the grammar, at the cost of using an ambiguous grammar that happens to be resolved deterministically by the conflict-resolution rule [Dragon §4.8.2].

#### Classic example 2: Operator precedence and associativity

The grammar `E → E + E | E * E | ( E ) | id` is ambiguous because `id + id * id` admits two parse trees: `(id + id) * id` and `id + (id * id)`. The fix is the structured G₀ above, which encodes precedence levels as nonterminal layers.

yacc/bison's `%left`, `%right`, `%nonassoc` declarations express the same information more compactly, allowing an ambiguous grammar to be used with the conflict resolved by the declared precedence/associativity [Dragon §4.8.2, EaC §3.4].

#### Inherent ambiguity

**Theorem (Parikh 1966):** The language L = {aⁱbʲcᵏ : i = j or j = k} is inherently ambiguous. Every CFG for L is ambiguous on strings of the form aⁿbⁿcⁿ, which have two parses — one from each clause of the definition. The proof uses Parikh's theorem to show the "mixing" of the two regular sublanguages creates unavoidable structural overlap [Hopcroft-Ullman, *Formal Languages and Their Relation to Automata*, 1969].

Inherent ambiguity is an exotic property in practice. No programming language designed for machine parsing is inherently ambiguous; the designer controls the grammar. But it is important theoretically: it shows that "choose an unambiguous grammar" is not always possible for every language you might want to describe.

### Normal forms

**Chomsky Normal Form (CNF) [Dragon §4.4.2]:** A CFG is in CNF if every production has one of two forms:
- A → BC (two nonterminals)
- A → a (one terminal)

Any CFG (without ε-productions at non-start symbols) can be converted to CNF by: (1) eliminating ε-productions, (2) eliminating unit productions A → B, (3) introducing new nonterminals for long right-hand sides and mixed terminal/nonterminal right-hand sides.

CNF matters primarily because the **CYK algorithm** (§7.5) requires it. CNF also makes inductive proofs about CFGs cleaner because every production has a simple binary structure.

**Greibach Normal Form (GNF):** Every production has the form A → aα where a ∈ T and α ∈ V*. Every CFG can be converted to GNF. GNF guarantees that every production consumes exactly one terminal, which means any string of length n requires exactly n production applications — a useful property for parsing-complexity analysis. GNF is the CFG analogue of the DFA's "one transition per input symbol."

## 7.2 Top-Down Parsing: Recursive Descent, FIRST/FOLLOW, LL(1)

Top-down parsing builds the parse tree from the root downward, always expanding the leftmost nonterminal. The parser must decide, at each expansion step, which production to apply. In predictive parsing, this decision is made by examining the next few tokens of input — without backtracking.

### Recursive descent: the mutual recursion of nonterminals

**Recursive descent** is the direct implementation of top-down parsing as a set of mutually recursive functions, one per nonterminal [EaC §3.3]. Each function attempts to match the right-hand side of one of the nonterminal's productions against the current input position.

The simplest variant — **backtracking recursive descent** — tries each alternative in order, backing up on failure:

```
function parseE():
    save = cursor
    if try_parse_E_plus_T():
        return success
    cursor = save
    if try_parse_T():
        return success
    return failure
```

Backtracking makes the parser correct but potentially exponential. The industrial-strength variant is **predictive recursive descent**, which uses the next token to choose the right production without backtracking:

```
function parseE():
    // E → T E'   (after left-recursion elimination — see §7.2.4)
    t = parseT()
    e_prime = parseE_prime()
    return node(E, t, e_prime)

function parseE_prime():
    if next_token == '+':
        consume('+')
        t = parseT()
        e_prime = parseE_prime()
        return node(E_prime, '+', t, e_prime)
    else:
        return ε_node()    // E' → ε applies
```

Predictive recursive descent works when the grammar is LL(1). Determining whether a grammar is LL(1) requires computing FIRST and FOLLOW sets.

### FIRST and FOLLOW sets

**Definition — FIRST(α) [Dragon §4.4.2]:** For any string α ∈ (V ∪ T)*, FIRST(α) is the set of terminals that can begin a string derived from α, plus ε if α ⟹* ε.

```
FIRST(ε) = {ε}
FIRST(a) = {a}                for terminal a
FIRST(A) = ⋃ { FIRST(α) : A → α ∈ P }   (fixed-point computation)
FIRST(Xα) = (FIRST(X) \ {ε}) ∪ (FIRST(α) if ε ∈ FIRST(X) else ∅)
```

**Definition — FOLLOW(A) [Dragon §4.4.3]:** For nonterminal A ∈ V, FOLLOW(A) is the set of terminals (including the end-marker $) that can appear *immediately to the right* of A in any sentential form:

```
FOLLOW(S) ⊇ {$}                              (S is the start symbol)
For each production B → αAβ:
    FOLLOW(A) ⊇ FIRST(β) \ {ε}
    if ε ∈ FIRST(β): FOLLOW(A) ⊇ FOLLOW(B)
```

Both computations are standard fixed-point iterations that converge because the sets only grow and the number of possible terminals is finite.

**Worked example for G₀ (after left-recursion elimination — grammar G₁ below):**

To compute FIRST and FOLLOW, we first eliminate left recursion from G₀ to obtain G₁:

```
G₁:
    E  →  T E'
    E' →  + T E'  |  ε
    T  →  F T'
    T' →  * F T'  |  ε
    F  →  ( E )   |  id
```

| Nonterminal | FIRST | FOLLOW |
|---|---|---|
| E  | { (, id } | { ), $ } |
| E' | { +, ε }  | { ), $ } |
| T  | { (, id } | { +, ), $ } |
| T' | { *, ε }  | { +, ), $ } |
| F  | { (, id } | { *, +, ), $ } |

**Derivation of FOLLOW(E'):** E appears only in the production E → T E'. Thus FOLLOW(E') ⊇ FOLLOW(E). E' appears in E' → + T E', giving FOLLOW(E') ⊇ FOLLOW(E'). Both contributions give FOLLOW(E') = FOLLOW(E) = {), $}.

**Derivation of FOLLOW(T'):** T appears in E → T E'. FIRST(E') = {+, ε}, so FOLLOW(T) ⊇ {+}. Since ε ∈ FIRST(E'), FOLLOW(T) ⊇ FOLLOW(E) = {), $}. T' appears in T → F T' and T' → * F T', so FOLLOW(T') ⊇ FOLLOW(T) = {+, ), $}.

### The LL(1) condition and parse table construction

A grammar G is **LL(1)** if for every nonterminal A and every pair of distinct productions A → α and A → β [Dragon §4.4.3]:

1. FIRST(α) ∩ FIRST(β) = ∅ (the two alternatives start with different terminals)
2. If ε ∈ FIRST(α): FOLLOW(A) ∩ FIRST(β) = ∅ (when α can derive ε, β's FIRST must not overlap with what follows A)

Equivalently: the **LL(1) parse table** M[A, a], defined as the set of productions that apply when nonterminal A is being expanded and terminal a is the current lookahead, has at most one entry in every cell. The table is constructed as:

```
For each production A → α:
    For each terminal a ∈ FIRST(α) \ {ε}:
        add A → α to M[A, a]
    If ε ∈ FIRST(α):
        For each terminal b ∈ FOLLOW(A):
            add A → α to M[A, b]
```

**LL(1) parse table for G₁:**

| | **id** | **+** | **\*** | **(** | **)** | **$** |
|---|---|---|---|---|---|---|
| **E**  | E→TE' |   |   | E→TE' |   |   |
| **E'** |   | E'→+TE' |   |   | E'→ε | E'→ε |
| **T**  | T→FT' |   |   | T→FT' |   |   |
| **T'** |   | T'→ε | T'→*FT' |   | T'→ε | T'→ε |
| **F**  | F→id |   |   | F→(E) |   |   |

Every cell has at most one entry — G₁ is LL(1). The cells for `E[id]`, `E[(` get `E→TE'` because `id,(` ∈ FIRST(T) = FIRST(F). The cells for `E'[)]` and `E'[$]` get `E'→ε` because those are in FOLLOW(E') and ε ∈ FIRST(E').

### The stack-based LL(1) parsing algorithm

```
LL1_parse(input w, grammar G₁, table M):
    stack = [S, $]                 // $ is bottom-of-stack sentinel
    input = w + [$]
    ip = 0                         // input pointer

    loop:
        X = stack.top(); a = input[ip]
        if X == $ and a == $: ACCEPT
        if X ∈ T ∪ {$}:
            if X == a: stack.pop(); ip++      // match terminal
            else: ERROR
        else:  // X is a nonterminal
            if M[X, a] = X → Y₁Y₂…Yₙ:
                stack.pop()
                stack.push(Yₙ, …, Y₁)       // push in reverse
            else: ERROR
```

**Trace for `id + id * id`:**

| Stack (top left) | Input remaining | Action |
|---|---|---|
| E $ | id + id * id $ | M[E,id] = E→TE' |
| T E' $ | id + id * id $ | M[T,id] = T→FT' |
| F T' E' $ | id + id * id $ | M[F,id] = F→id |
| id T' E' $ | id + id * id $ | match id |
| T' E' $ | + id * id $ | M[T',+] = T'→ε |
| E' $ | + id * id $ | M[E',+] = E'→+TE' |
| + T E' $ | + id * id $ | match + |
| T E' $ | id * id $ | M[T,id] = T→FT' |
| F T' E' $ | id * id $ | M[F,id] = F→id |
| id T' E' $ | id * id $ | match id |
| T' E' $ | * id $ | M[T',*] = T'→*FT' |
| * F T' E' $ | * id $ | match * |
| F T' E' $ | id $ | M[F,id] = F→id |
| id T' E' $ | id $ | match id |
| T' E' $ | $ | M[T',$] = T'→ε |
| E' $ | $ | M[E',$] = E'→ε |
| $ | $ | ACCEPT |

### Eliminating left recursion

G₀ is not LL(k) for any k because it is **left-recursive**: E → E + T has E on both left and right. Left recursion causes infinite loops in recursive descent.

**Direct left recursion elimination [Dragon §4.3.3]:** For a nonterminal A with productions A → Aα₁ | Aα₂ | … | β₁ | β₂ | … (where no βᵢ begins with A):

```
A   →  β₁ A'  |  β₂ A'  |  …
A'  →  α₁ A'  |  α₂ A'  |  ε
```

For E → E + T | T: β₁ = T, α₁ = +T, giving E → T E', E' → +T E' | ε.
For T → T * F | F: β₁ = F, α₁ = *F, giving T → F T', T' → *F T' | ε.
F has no left recursion.

**Indirect left recursion** (A → Bα, B → Aβ) requires a more general algorithm: topologically order the nonterminals, then substitute earlier nonterminals into later ones until only direct left recursion remains, then apply the direct elimination transform [Dragon §4.3.3, EaC §3.3.1].

### Left factoring

Left recursion addresses one LL(1) failure mode; **left factoring** addresses the other: common prefixes across alternatives. If A → αβ₁ | αβ₂, then on seeing the first token of α the parser cannot decide which alternative to use. The fix:

```
A   →  α A'
A'  →  β₁  |  β₂
```

For example, the grammar fragment `stmt → if E then stmt | if E then stmt else stmt` has a common prefix `if E then stmt`. After left factoring: `stmt → if E then stmt stmt_tail` and `stmt_tail → else stmt | ε`. This is exactly the left-factored form underlying the dangling-else grammar.

### LL(k) for k > 1 and LL(*)

**LL(k) grammars** allow k tokens of lookahead in the parse table. The table has entries indexed by (nonterminal, sequence of k tokens). The LL(k) table can be exponentially larger than LL(1): with |T| terminals and k lookahead, the table has |V| × |T|^k entries. For k = 3 and |T| = 100, this is 100 × 10⁶ entries — impractical for most tools.

More importantly, some grammars are not LL(k) for any fixed k but can be parsed deterministically with an unbounded lookahead search. Terence Parr's **LL(*) analysis** (2011) addresses this [Parr, "LL(*): The Foundation of the ANTLR Parser Generator," *PLDI 2011*]. The key insight: instead of a finite lookahead bound, build a DFA over the remaining input to decide which alternative to take. The prediction DFA for a non-deterministic choice can be constructed offline (at grammar analysis time) and run online during parsing. When the prediction DFA reaches an accepting state, it has determined which alternative to take.

**ALL(*) (Adaptive LL(*))** [Parr, Harwell, Fisher, "Adaptive LL(*) Parsing," *OOPSLA 2014*] extends LL(*) to grammars that are not LL(*) either. Instead of an offline DFA, ALL(*) builds the prediction automaton *adaptively* during parsing: when the parser encounters a prediction choice, it uses a parallel simulation of all alternatives, caching the result as a DFA for future occurrences of the same choice at the same grammar state. Inputs that trigger ambiguity cause an error (or a configurable conflict resolution). ANTLR 4 implements ALL(*); it handles the vast majority of real-world grammars including Java, Python, SQL, and most DSLs.

## 7.3 Bottom-Up Parsing: LR(0), SLR, LALR, LR(1), GLR

Bottom-up parsing builds the parse tree from the leaves upward by identifying the **handle** of a right-sentential form and reducing it to the corresponding nonterminal. The approach is more powerful than top-down: every LL(k) grammar is also LR(k), but the converse fails badly. G₀ — left-recursive, not LL(1) — is LR(1) without any transformation.

### Handles and the shift-reduce paradigm

**Right-sentential form:** A sentential form reachable by rightmost derivation from S.

**Handle:** In a right-sentential form αβw, the handle is the production A → β together with its position such that αAw ⟹_rm αβw. Intuitively, the handle is the piece of the current string that should be reduced next [Dragon §4.5.1].

The **parsing stack** maintains the portion of the input that has been shifted but not yet reduced. The algorithm repeatedly applies one of two operations:

- **Shift:** Move the next input token onto the stack.
- **Reduce:** The top of the stack matches the right-hand side of some production A → β; pop β, push A.

The parser accepts when the stack contains only the start symbol and the input is exhausted. The challenge is deciding when to shift versus reduce — this is the fundamental decision that LR theory solves via automata.

### LR(0) items and the canonical collection

An **LR(0) item** is a production with a dot at some position in its right-hand side, e.g., E → E · + T. The dot marks "how far we have gotten" — to the left of the dot is on the stack; to the right remains to be parsed. An item E → E + · T says: "I have an E and a + on the stack; I now expect a T."

**The augmented grammar G':** Add a new production S' → S$ to mark the accept state. For G₀, the augmented grammar adds E' → E$ (using $ as the end-marker, distinct from the existing nonterminals).

**Closure:** Given a set of items I, closure(I) extends I with all items for nonterminals whose parsing must begin from the current position:

```
closure(I):
    J = I
    while change:
        for each item [A → α · B β] in J:
            for each production B → γ:
                if [B → · γ] ∉ J: J ∪= {[B → · γ]}
    return J
```

**Goto:** Given a set of items I and a grammar symbol X (terminal or nonterminal), goto(I, X) is the closure of items whose dot is immediately after X:

```
goto(I, X) = closure({ [A → αX · β] : [A → α · X β] ∈ I })
```

**The canonical LR(0) collection** is the set of all item sets reachable from the start item via goto transitions. For G₀ (augmented as E' → · E$), the canonical collection has 12 states [Dragon §4.6.2]. We label states I₀ through I₁₁:

```
I₀ = closure({E' → · E$}) =
    { E' → · E$,
      E  → · E + T,   E  → · T,
      T  → · T * F,   T  → · F,
      F  → · ( E ),   F  → · id }

goto(I₀, E) = I₁ = closure({E' → E · $, E → E · + T}) =
    { E' → E · $, E → E · + T }

goto(I₀, T) = I₂ = closure({E → T ·, T → T · * F}) =
    { E → T ·, T → T · * F }

goto(I₀, F) = I₃ = { T → F · }

goto(I₀, () = I₄ = closure({F → ( · E )}) =
    { F → ( · E ), E → · E + T, E → · T,
      T → · T * F, T → · F, F → · ( E ), F → · id }

goto(I₀, id) = I₅ = { F → id · }

goto(I₁, +) = I₆ = closure({E → E + · T}) =
    { E → E + · T, T → · T * F, T → · F,
      F → · ( E ), F → · id }

goto(I₂, *) = I₇ = closure({T → T * · F}) =
    { T → T * · F, F → · ( E ), F → · id }

goto(I₄, E) = I₈ = closure({F → ( E · ), E → E · + T}) =
    { F → ( E · ), E → E · + T }

goto(I₆, T) = I₉ = closure({E → E + T ·, T → T · * F}) =
    { E → E + T ·, T → T · * F }

goto(I₇, F) = I₁₀ = { T → T * F · }

goto(I₈, )) = I₁₁ = { F → ( E ) · }
goto(I₈, +) = I₆   (same as goto(I₁, +))
goto(I₄, F) = I₃   goto(I₄, () = I₄   goto(I₄, id) = I₅
goto(I₆, F) = I₃   goto(I₆, () = I₄   goto(I₆, id) = I₅
goto(I₇, () = I₄   goto(I₇, id) = I₅
goto(I₉, *) = I₇
```

### SLR(1): FOLLOW-set disambiguation

**SLR(1)** (Simple LR) uses the LR(0) automaton but resolves shift-reduce conflicts using FOLLOW sets [Dragon §4.7.1]. The parsing action table is:

- **Shift** on token a from state Iᵢ if goto(Iᵢ, a) = Iⱼ exists.
- **Reduce** A → β from state Iᵢ if Iᵢ contains A → β · *and* a ∈ FOLLOW(A).
- **Accept** in state containing E' → E · $.

For state I₂ = {E → T ·, T → T · * F}: there is a shift on `*` (to I₇) and a potential reduce on E → T. The reduce fires only for tokens in FOLLOW(E) = {+, ), $}. No conflict: shift takes `*`, reduce takes `+`, `)`, `$`. G₀ is SLR(1).

SLR(1) is weaker than full LR(1) because FOLLOW(A) is a global approximation: it includes all contexts where A can appear, not just the contexts reachable from the current parser state. For grammars where this approximation introduces spurious conflicts, SLR(1) fails but LR(1) succeeds.

### LALR(1): merging LR(1) states with identical cores

**LR(1) items** carry an explicit lookahead: [A → α · β, a] means "I have α on the stack, expecting β from input, and if this production is applied, the next token will be a" [Dragon §4.7.2]. The LR(1) canonical collection has one item set per (stack configuration, lookahead context) pair — exponentially more states than LR(0) for grammars with many contexts.

**LALR(1)** (Look-Ahead LR) observes that many LR(1) item sets have identical **cores** — the same LR(0) items, differing only in their attached lookahead sets. LALR merges all LR(1) states with the same core, unioning their lookahead sets [Dragon §4.7.4, EaC §3.5]. The result has the same number of states as the LR(0) automaton.

**The LALR table construction algorithm:**
1. Build the canonical LR(1) collection.
2. Group states by their core (ignoring lookaheads).
3. For each group, merge the lookaheads: [A → α · β, {a₁,…}] in one state and [A → α · β, {b₁,…}] in another merge to [A → α · β, {a₁,…,b₁,…}].
4. Rebuild the goto table on the merged states (goto is determined by the core, which is unchanged).
5. Check for conflicts in the merged action table.

**The efficiency advantage:** The canonical LR(1) collection for G₀ has the same 12 states as the LR(0) collection (G₀ is simple enough that every LR(1) state has a distinct core). For grammars with many nonterminals, LR(1) may have thousands of states that LALR reduces to hundreds. yacc, bison, and virtually all production parser generators use LALR(1) [Dragon §4.9].

**LALR(1) for G₀:** FOLLOW sets for G₀ match the LR(1) lookaheads for this grammar, so the LALR(1) table is identical to the SLR(1) table. We show the action/goto tables:

**Action table (partial, key states):**

### LR(1): when LALR fails

Merging LR(1) states with identical cores is safe *unless* the union of lookahead sets introduces a reduce-reduce conflict that was absent in the original LR(1) states. The canonical discriminating grammar is [Dragon §4.7.4]:

```
S'→ S
S → a A d  |  b B d  |  a B e  |  b A e
A → c
B → c
```

The LR(1) collection has two states whose core is {A→c·, B→c·} but with disjoint lookahead sets: one state has {A→c·, d} and {B→c·, e}; the other has {A→c·, e} and {B→c·, d}. Merging them gives a state with {A→c·, d/e} and {B→c·, d/e} — a reduce-reduce conflict (both A→c and B→c can reduce on d or e). The language is deterministic, the LR(1) grammar correctly handles it, but LALR fails.

For this reason, some programming language grammars (particularly those derived from natural language descriptions without careful disambiguation) are LR(1) but not LALR(1). In practice this is rare for well-designed programming languages, which is why LALR(1) is sufficient for most parser generators [EaC §3.4].

### GLR: Generalized LR and Tomita's algorithm

**GLR (Generalized LR)** [Tomita 1985, *Efficient Parsing for Natural Language*, Kluwer] extends LR parsing to handle ambiguous grammars by maintaining *all* possible parser states simultaneously rather than requiring deterministic resolution of conflicts.

The key data structure is the **Graph-Structured Stack (GSS)** — a directed acyclic graph that compactly represents the exponentially many parse stacks that would arise from naive parallelism. Each node in the GSS is a pair (state, subtree); edges represent stack links. When a shift-reduce conflict arises, both the shift and reduce paths are pursued in parallel, sharing stack prefixes via the GSS.

**Algorithm sketch:**

```
GLR_parse(input tokens):
    active_stacks = GSS containing only the initial state
    for each token t in input:
        // Reducer phase: apply all possible reductions
        for all active states s with a reduction on t:
            reduce on the stack, creating new GSS nodes
            add new states to active_stacks
        // Shifter phase: shift t on all resulting stacks
        for all active states s with a shift on t:
            shift t, creating new GSS nodes
    accept if any active stack ends in the accept state
```

The GSS allows sharing: if two stacks reach the same state after the same prefix, they merge to a single node. This sharing bounds the GSS to O(n²) nodes for ambiguous grammars (where n is the input length), giving O(n³) overall parsing time — the same as the Earley algorithm (§7.5).

**Applications:**

- **Natural language processing:** NLP grammars are highly ambiguous; GLR handles them while returning the forest of all parse trees (or a packed parse forest).
- **C++ parsing:** The C++ grammar is notoriously ambiguous at the declaration/expression level and depends on type information not available during parsing (`A * B` is either multiplication or a pointer declaration). Several C++ parsing tools (notably the Elkhound parser and the KCC semantics) use GLR to handle the grammar ambiguity directly.
- **Island parsing:** Parsing fragments of languages (e.g., extracting function signatures from arbitrary source code) can use GLR to ignore non-matching input.

GLR is not practical for interactive compilation — O(n³) with large constants is too slow for a compiler's main loop. It is a research and tooling technique, not a production compiler technique.

## 7.4 Operator-Precedence and Pratt Parsing

Expression parsing occupies a disproportionate fraction of a compiler's hot path. LALR handles expressions correctly via the multilevel nonterminal hierarchy (E, T, F in G₀), but the grammar is verbose and each operator requires a new nonterminal level. Pratt parsing provides a cleaner, more efficient alternative that handles all binary operators in a single function.

### Pratt's top-down operator precedence

**Top-down operator precedence (TDOP)**, introduced by Vaughan Pratt at POPL 1973 [Pratt, "Top down operator precedence," *POPL 1973*], structures expression parsing around two properties of each token:

- **Null denotation (nud):** How to parse a token that appears at the start of an expression (prefix position). For `id`, nud returns the identifier node. For `-`, nud parses a unary negation.
- **Left denotation (led):** How to parse a token that appears *after* an expression (infix position). For `+`, led takes the left operand and parses the right. The binding power of the token controls precedence.

**Binding power (bp):** Each infix operator has a left binding power (lbp) and right binding power (rbp). For left-associative operators, rbp = lbp. For right-associative operators, rbp = lbp - 1. Higher binding power means tighter binding (higher precedence).

```python
def parse_expression(min_bp=0):
    t = consume()
    left = t.nud()                    # handle prefix/atom
    while True:
        t = peek()
        if t.lbp <= min_bp: break     # stop if binding is too loose
        t = consume()
        left = t.led(left)            # handle infix/postfix
    return left
```

For `+` with lbp=10, rbp=10 (left-associative) and `*` with lbp=20, rbp=20:

```
parse_expression(0):
    left = id("a")                    # nud of id
    peek is +, lbp=10 > 0: consume +
    left = +(a, parse_expression(10)) # led of +, min_bp=10
        parse_expression(10):
            left = id("b")
            peek is *, lbp=20 > 10: consume *
            left = *(b, parse_expression(20))
                parse_expression(20):
                    left = id("c")
                    peek is $, lbp=0 ≤ 20: stop
                    return c
            left = *(b, c)
            peek is $, lbp=0 ≤ 10: stop
            return *(b, c)
    left = +(a, *(b, c))
    peek is $, lbp=0 ≤ 0: stop
    return +(a, *(b, c))
```

Result: `a + b * c` → `+(a, *(b, c))` — correct with standard precedence.

**Precedence climbing** is an equivalent algorithm with a slightly different formulation: the recursive call carries the minimum allowed binding power rather than the right binding power. The two algorithms are isomorphic [Crockford 2007, Norvell 1999].

**Why Pratt > LALR for expressions:**
1. Single function, O(n) time without a parser generator
2. Precedences are just integers — trivially configurable at runtime (important for languages with user-defined operators)
3. Straightforward extension to ternary (`?:`), mixfix, and prefix operators via nud/led specialization
4. Error messages are cleaner because the call stack reflects the expression structure

Many expression parsers in production compilers (including Clang's, for certain subexpressions) use a variant of Pratt parsing for their expression sub-grammar, even when the rest of the grammar is handled by recursive descent [Chapter 32].

## 7.5 Earley Parsing and CYK

### Earley's algorithm

Earley parsing [Earley, "An efficient context-free parsing algorithm," *CACM 1970*] recognizes any CFG in O(n³) time and O(n²) space, with better bounds for restricted grammar classes. It is the most general efficient parsing algorithm — it handles ambiguous grammars, left recursion, ε-productions, and all CFGs without preprocessing [EaC §3.7].

**Earley items:** An Earley item is a triple [A → α · β, j] where A → αβ is a production, the dot separates the portion already matched (α) from what remains (β), and j is the **origin** — the position in the input where this match began.

**The chart:** Earley's algorithm maintains an array of sets S[0..n] where S[i] is the set of Earley items whose right context ends at position i.

**Three operations:**

```
Predictor([A → α · B β, j], i):
    // B is a nonterminal; predict all B-productions starting at position i
    For each production B → γ:
        add [B → · γ, i] to S[i]

Scanner([A → α · a β, j], i):
    // a is a terminal; if input[i] = a, advance the dot
    If input[i] = a:
        add [A → α a · β, j] to S[i+1]

Completer([B → γ ·, j], i):
    // B-production is complete at position i, starting from j
    For each [A → α · B β, k] in S[j]:
        add [A → α B · β, k] to S[i]
```

**Algorithm:**

```
Earley_parse(input a₁a₂…aₙ, grammar G):
    S[0] = closure_Predictor({[S' → · S, 0]})
    for i = 0 to n:
        for each item in S[i]:                  // worklist, may grow during loop
            if item = [A → α · B β, j] and B ∈ V: Predictor
            elif item = [A → α · a β, j] and a ∈ T: Scanner → adds to S[i+1]
            elif item = [A → γ ·, j]:           Completer
    Accept if [S' → S ·, 0] ∈ S[n]
```

**Complexity analysis [EaC §3.7]:**
- **General CFG:** O(n³) time and O(n²) space. The cubic term comes from the Completer: for each completed item at position i with origin j, we scan all items in S[j], which can be O(n) items, and we have O(n²) completed items.
- **Unambiguous grammar:** O(n²) time. When the grammar is unambiguous, each position generates O(n) items total, reducing the Completer work to O(n²).
- **LR(k) grammar:** O(n) time. Leo's 1991 optimization [Leo, "A general context-free parsing algorithm running in linear time on every LR(k) grammar," *Theoretical Computer Science 1991*] detects right-recursive structures and collapses chains of completions into a single pointer, bringing Earley performance to linear on LR(k) grammars.

Earley parsing is rarely used in production compilers (LR tools are faster and well-integrated with code generation), but it is essential in tools that need to parse arbitrary CFGs — grammar workbenches, natural language processors, and the NLTK chart parser.

### CYK (Cocke-Younger-Kasami)

**CYK** [Cocke 1969, Younger 1967, Kasami 1965] is a dynamic programming algorithm for CFG recognition that requires the grammar to be in **CNF** [Dragon §4.4.2].

**CYK table:** For input a₁a₂…aₙ, maintain a table T[i][j] (1 ≤ i ≤ j ≤ n) where T[i][j] is the set of nonterminals that can derive the substring aᵢ…aⱼ.

**Fill the table:**

```
CYK(input a₁…aₙ, grammar G in CNF):
    for i = 1 to n:
        T[i][i] = { A : A → aᵢ ∈ P }     // base case: single terminals

    for l = 2 to n:                        // l = span length
        for i = 1 to n-l+1:
            j = i + l - 1
            T[i][j] = ∅
            for k = i to j-1:             // split point
                for each A → B C ∈ P:
                    if B ∈ T[i][k] and C ∈ T[k+1][j]:
                        T[i][j] ∪= {A}

    Accept if S ∈ T[1][n]
```

**Complexity:** O(n³ |G|) time and O(n² |V|) space. The three nested loops (span, start, split) give the cubic term; the innermost loop scans all CNF productions.

**Applications:**
- **Ambiguity counting:** T[i][j] records all possible parses of each substring; counting the number of times S appears in T[1][n] (using a weighted variant) gives the total number of parse trees — useful for ambiguity analysis.
- **NLP:** PCFG parsing (probabilistic context-free grammars) is a straightforward extension of CYK where each grammar rule carries a probability weight and the algorithm computes the most-probable parse [Charniak, *Statistical Language Learning*, 1993].
- **Compilers (rarely):** CYK is not used in production compilers — the CNF requirement doubles grammar size, the O(n³) constant is large, and LALR(1) is linear for all grammars that practical languages use. CYK appears in compiler courses as a theoretical reference point.

## 7.6 PEG and Packrat Parsing

### Parsing Expression Grammars

**PEG (Parsing Expression Grammars)** [Ford, "Parsing Expression Grammars: A Recognition-Based Syntactic Foundation," *POPL 2004*] replaces the nondeterministic choice of CFGs with **ordered choice**: instead of A → α | β (either α or β, possibly both yielding parses), PEG uses A ← α / β meaning "try α; if it fails, try β." This makes PEGs always unambiguous — ordered choice resolves every conflict in favor of the first matching alternative.

**PEG operators:**

| Operator | Meaning |
|---|---|
| e₁ e₂ | Sequence: match e₁ then e₂ |
| e₁ / e₂ | Ordered choice: try e₁; if it fails at *this* position, try e₂ |
| e* | Zero or more: greedy repetition of e |
| e+ | One or more: e e* |
| e? | Optional: e / ε |
| &e | And-predicate: succeeds without consuming if e would succeed |
| !e | Not-predicate: succeeds without consuming if e would fail |

The `!e` operator is particularly powerful: `!keyword id` matches an identifier that is not a keyword, a pattern that CFGs require careful grammar engineering to express.

**PEG vs. CFG alternation:** CFG alternation is set-theoretic union — both alternatives contribute to the language. PEG ordered choice is sequential — if the first alternative succeeds, the second is never tried. This means the PEG `A ← a / a b` always matches the single `a`, never `a b` — a property called the **left-anchoring commitment**. The CFG `A → a | a b` is ambiguous on input `a b` (both `a` alone and `a b` are valid matches), but the PEG is not.

**Are PEGs more expressive than CFGs?** The relationship is subtle. Some languages are PEG-recognizable but not context-free (e.g., `{aⁿbⁿcⁿ}` is recognizable by PEG with predicates). Some context-free languages may not have direct PEG analogues without left recursion transformations. Left recursion in PEGs causes infinite loops and requires special handling [Warth, Douglass, Millstein, "Packrat Parsers Can Support Left Recursion," *PEPM 2008*].

### Packrat parsing

**Packrat parsing** [Ford, "Packrat Parsing: Simple, Powerful, Lazy, Linear Time," *ICFP 2002*] achieves linear-time parsing of any PEG via **memoization**: for each (rule, position) pair, store the result of applying the rule at that position. On a second attempt to apply the same rule at the same position, return the stored result without repeating the work.

**Memoization table:** A 2D table `memo[rule][position]` of size O(|G| × n), where |G| is the number of rules and n is the input length. Each entry stores the result of applying a rule at a position: either a failure marker or the (result, consumed_length) pair.

```
parse(rule, pos):
    if memo[rule][pos] is set:
        return memo[rule][pos]     // cache hit — O(1)
    result = apply_rule(rule, pos) // may recurse
    memo[rule][pos] = result
    return result
```

**Linearity:** Each (rule, position) pair is computed at most once. There are O(|G| × n) pairs. Each computation does a constant amount of work assuming sub-computations are cached. Total: O(|G| × n) = O(n) for fixed grammar.

**The space tradeoff:** Linear time comes at O(|G| × n) space — potentially large. For a grammar with 100 rules and a 100,000-character input, the memoization table has 10,000,000 entries. Each entry may store a tree node pointer — 8 bytes on a 64-bit system — giving 80 MB per file. In practice, streaming or chunk-based packrat parsing releases memo entries for positions well behind the current parse point, reducing space at the cost of some re-computation on backtracking.

**Interaction with side effects:** Packrat parsing assumes that parsing rules are pure functions of their (rule, position) input. Rules with side effects (updating a symbol table, emitting code) break the memoization invariant — a cached result from one call may omit side effects expected by a second call from a different context. Languages with complex semantic actions (C's context-dependent declarations) cannot use packrat directly.

## 7.7 Parser Combinators

**Parser combinators** express a grammar as a set of higher-order functions that can be composed algebraically. The type-theoretic view, originating in Haskell research [Hutton, "Higher-Order Functions for Parsing," *JFP 1992*; Leijen, Meijer, "Parsec: Direct Style Monadic Parser Combinators for the Real World," *Haskell Workshop 2001*]:

```haskell
type Parser a = String → [(a, String)]
-- A parser of a's takes a string, returns a list of (result, remaining-input) pairs.
-- Empty list: parse failure. Multiple elements: ambiguous parses.
```

**Core combinators:**

```haskell
-- Sequence
seq :: Parser a → Parser b → Parser (a, b)
seq p q input = [(( x, y), s₂) | (x, s₁) ← p input, (y, s₂) ← q s₁]

-- Alternative (non-deterministic)
alt :: Parser a → Parser a → Parser a
alt p q input = p input ++ q input

-- Many (zero-or-more, greedy)
many :: Parser a → Parser [a]
many p = alt (seq p (many p)) (return [])

-- Many1 (one-or-more)
many1 p = seq p (many p)

-- Optional
optional p = alt (fmap Just p) (return Nothing)

-- Return (always succeeds, consumes nothing)
return x input = [(x, input)]

-- Item (consume one character)
item [] = []; item (c:cs) = [(c, cs)]
```

**Monadic parsing:** The list monad makes combinators compose naturally:

```haskell
expr  = do { t ← term; e' ← expr'; return (T t e') }
term  = do { f ← factor; t' ← term'; return (T' f t') }
```

This is exactly the recursive-descent parser for G₁, expressed at a higher level of abstraction. The `do`-notation unfolds to `seq` and `>>=` operations.

**Industrial parser combinator libraries:**

- **Parsec/Megaparsec (Haskell):** The foundational production library. Megaparsec extends Parsec with better error messages, Unicode support, and `try` for backtracking combinators. Used in GHC's parser for non-performance-critical code generation paths.
- **attoparsec (Haskell):** A zero-copy, focus on throughput. Used in high-performance protocol parsing (e.g., Snap HTTP server). No backtracking in the LL sense — commits on first character of each alternative.
- **nom (Rust):** Macro-based combinator library with zero-copy parsing of binary protocols. No dynamic dispatch; combinators compose via generic types.
- **pest (Rust):** PEG-based combinators with packrat memoization.

**Limitations:**

1. **Left recursion:** Direct left recursion causes infinite loops. Indirect left recursion requires transformation or a specialized combinator. Megaparsec provides no support; nom requires manual left-factoring.

2. **Error messages:** The default error in a combinator library says "expected token X at position Y" — helpful, but not as informative as a hand-written parser that knows it is inside a function declaration. The Parsec error model tracks the deepest failure point; Megaparsec refines this with labeled alternatives.

3. **Performance:** The polymorphic dispatch and list-returning semantics of pure combinators impose overhead. Attoparsec and nom mitigate this by using callbacks, CPS, or continuation-passing style rather than list returns. A typical attoparsec parser runs at 100–500 MB/s — competitive with re2c for simple grammars.

Parser combinators occupy a middle ground: more flexible than generated parsers (no separate grammar file, grammar embedded in the host language), more maintainable than raw recursive descent (algebraic composition), but slower than hand-optimized parsers and weaker in error recovery.

## 7.8 Error Recovery Strategies

A production compiler must continue parsing after encountering a syntax error to discover subsequent errors in a single compilation. Error recovery is partly formal, partly heuristic, and always a source of false positives in the diagnostic output.

### Panic mode

**Panic mode** discards input tokens until a **synchronization token** is found — a token likely to be at a clean boundary in the grammar (`;`, `}`, `end`, `begin`, etc.) [Dragon §4.4.5]. The parser then resets to a known state.

```
on error in recursive_descent:
    while next_token ∉ sync_set:
        consume_token()
    // resume from known state
```

Panic mode is simple, robust, and crude. It can skip entire sub-expressions, causing many follow-on errors to be suppressed (good) or genuine errors to be missed (bad). C compilers typically use `;` and `}` as synchronization tokens; expression parsers use `,` and `)`.

### Phrase-level recovery

**Phrase-level recovery** makes a local correction — inserting or deleting a single token — and continues parsing. If the parser expected `;` and sees `}`, it inserts a virtual `;` token. If it sees `;;`, it deletes one.

This can cascade: inserting a token may cause subsequent tokens to fail, leading to further insertions. Compilers like GCC implement phrase-level recovery with heuristics accumulated over decades: "if a function body is missing its return statement, do not insert one because the type may be void." Such heuristics are not formalizable but are invaluable in practice.

### Error productions

**Error productions** add explicit grammar rules for common mistakes. For example:

```
stmt  →  if expr then stmt else stmt
       |  if expr then stmt            // correct
       |  if expr stmt                 // common error: missing 'then'
```

The error production for the missing `then` fires when the parser sees `stmt` where `then` was expected, allowing a specific diagnostic: "expected 'then' after condition." GCC and Clang have dozens of such productions implicitly encoded in their hand-written parsers.

Error productions require that the grammar author anticipate specific mistakes — they do not generalize automatically. The benefit is precise, actionable diagnostics for common mistakes, as opposed to the generic "unexpected token" of panic mode.

### Global correction

**Minimum-edit-distance error recovery** finds the smallest set of token insertions, deletions, and substitutions that transforms the erroneous input into a valid string [Aho-Peterson 1972, Graham-Harrison-Ruzzo 1980]. This is theoretically optimal in the sense that it minimizes disruption to the programmer's intent.

**Complexity:** O(n³) per error, where n is the input length — too slow for production use on large files. The algorithm also requires knowing the complete LR automaton, which limits its applicability to generated parsers. The only production use case has been in certain SQL parsers and language workbenches where correctness of recovery matters more than parsing speed.

### Synchronization in recursive descent

Hand-written recursive-descent parsers can implement a refined form of panic mode by passing a **follow set** (the set of tokens expected to follow the current construct) down through recursive calls. When an error is detected:

```python
def parse_statement(follow):
    expected = {SEMICOLON, RBRACE}
    if not parse_expression(expected | follow):
        # recover: skip to expected ∪ follow
        while next_token not in (expected | follow):
            consume()
    if next_token == SEMICOLON:
        consume()  # synchronize on semicolon
    else:
        error("missing semicolon", follow)
```

The follow set propagation ensures that the recovery point respects the nesting structure of the grammar. A missing `;` inside a loop body should not consume the `}` that closes the loop — that `}` is in the follow set passed from the enclosing statement parser.

Clang's parser uses exactly this pattern. Each `Parse*` function takes an implicit follow context (via the `ExpectAndConsumeSemi()` and `SkipUntil()` helper methods in `clang/lib/Parse/Parser.cpp`) and implements phrase-level and synchronization-based recovery tailored to each syntactic construct [Chapter 32 covers Clang's parser in detail].

## 7.9 Incremental Parsing for IDEs

Chapter 6 (§6.8) discussed incremental lexing; the IDE requirement extends to the parser level. A keystroke changes a few tokens; the IDE must produce a valid AST (or a maximally valid partial AST) within 50 ms. Full re-parsing of every edit is feasible for small files but fails for codebases where a single header file expands to hundreds of thousands of lines.

### The requirement

**Formal requirement:** Given a parse tree T for input w and an edit e (insertion, deletion, or replacement of a contiguous range of tokens), produce the parse tree T' for w' = apply(e, w) in time O(|T △ T'|) — proportional to the symmetric difference of the two trees, not the file size.

This is achievable only under the assumption that the grammar is **context-free in the sense that subtrees are reusable** — a subtree rooted at nonterminal A is valid regardless of the surrounding context. CFGs satisfy this by definition; context-sensitive features (type-dependent parsing in C++) break it.

### Tree-sitter's approach

**Tree-sitter** [Brunsfeld et al., 2018; https://tree-sitter.github.io] implements a robust incremental parser used by GitHub, Neovim, Helix, and dozens of other editor tools. Its design makes several key choices:

**LR parsing with retained parse tree:** Tree-sitter generates an LR(1)-like parser (using a variant of the canonical LR table). On each edit, it computes a minimal set of *retokenized* and *reparsed* regions:

1. **Relex the affected region:** Starting from the edit point, re-lex tokens until reaching a position where the new token stream matches the old one (same type and text). This uses the checkpoint-based incremental lexer described in §6.8.

2. **Reparse the affected subtrees:** Starting from the smallest subtree spanning the changed tokens, replay the LR automaton. If the LR state at the end of re-parsing matches the old state at the same position, the subtree above can be reused unchanged.

3. **Reuse subtrees by identity:** Unchanged subtrees are pointers to the old tree — no copying. The new tree shares structure with the old tree (a persistent data structure).

**Two-phase parsing:**

Tree-sitter uses a two-phase approach: an outer **syntax grammar** (LR-based, handling the tree structure) and an inner **lexical grammar** (DFA-based, context-dependent tokenization). Each LR state can have a different set of active lexical patterns — so that `<` is tokenized differently inside template argument lists than in comparison expressions.

**Error recovery:** Tree-sitter's error recovery uses a combination of:
- *Error nodes:* Invalid tokens are wrapped in `ERROR` nodes and parsing continues.
- *Lookahead-based skipping:* When the LR automaton is in an error state, the parser skips tokens until one of the "expected" tokens for the current state is found.
- *Partial trees:* The resulting parse tree always covers the entire input, with ERROR nodes marking invalid regions — essential for IDE use where the file is transiently invalid.

### The challenges

**Context-sensitivity:** C and C++ type declarations make the grammar context-sensitive at the expression level. Tree-sitter handles C++ by using an ambiguous grammar that allows both interpretations and uses a secondary analysis to resolve the ambiguity post-parse — a form of GLR at the tree level. Full semantic disambiguation is left to the LSP server.

**Ambiguity:** When the grammar is ambiguous and two parse trees are valid for a region, tree-sitter uses priority rules (analogous to yacc conflict resolution) to pick one deterministically. This can produce incorrect trees in rare cases, trading correctness for speed.

**Error recovery in incremental mode:** After an edit, the retained subtrees may have been parsed in error-recovery mode. Incremental re-parsing must propagate error states correctly — a re-parsed subtree that was previously in error recovery may now be valid, and the surrounding context must be updated.

### clangd's incremental architecture

As noted in §6.8, clangd does not implement character-level incremental parsing. Instead, it relies on two mechanisms:

1. **Preamble PCH:** The stable prefix of a translation unit (all `#include` headers) is compiled once to a precompiled header and reused across edits. This reduces the incremental parsing scope to just the user-visible file.

2. **Background indexing:** Semantic information (symbol locations, type information, cross-references) is maintained in a background index that is updated asynchronously. The foreground re-parse updates only the current file's AST; the background index catches up within a few seconds.

This coarser-grained incrementality is sufficient because Clang's parser is fast enough (a 1000-line implementation file re-parses in well under 50 ms) and correct (no approximations from subtree reuse).

The fundamental tradeoff: tree-sitter achieves very fast incremental updates (microseconds for single-character edits) at the cost of an approximate parse tree and limited semantic information. clangd achieves a precise, semantically correct AST at the cost of re-parsing the non-preamble portion of the file on every edit. For editor syntax highlighting, tree-sitter is the right choice; for type-aware diagnostics and refactoring, clangd's approach is required.

## 7.10 Why Production Compilers Use Hand-Written Recursive Descent

The parsing theory developed in §§7.2–7.6 offers a rich palette of tools: LL(1) table-driven parsers, LALR generators, GLR, Earley, PEG, combinator libraries. Yet every major production compiler for a complex language uses a **hand-written recursive-descent parser**:

| Compiler | Language | Parser strategy |
|---|---|---|
| Clang | C/C++/Obj-C | Hand-written recursive descent since 2007 |
| GCC | C/C++/Fortran | Hand-written recursive descent since 2004 (converted from YACC) |
| rustc | Rust | Hand-written recursive descent |
| Go compiler | Go | Hand-written recursive descent |
| Swift | Swift | Hand-written recursive descent |
| Kotlin | Kotlin | Hand-written recursive descent (via IntelliJ's parser framework) |

The reasons are both theoretical and pragmatic.

### C++ is not LL(1) or LR(k) for any k

The C++ grammar is not context-free. The classic example is the "most vexing parse": `T(x);` is either a declaration of a variable `x` of type `T` or a call to a function `T` with argument `x` — depending on whether `T` is a type name. This disambiguation requires consulting the symbol table, which is populated *during* parsing. A formal grammar-driven parser would need to interleave semantic analysis with parsing in a way that no grammar formalism captures cleanly.

Additionally, C++ templates introduce meta-level computation that can generate arbitrarily complex types, and instantiating templates during parsing (to determine whether template arguments are types or values) is known to be undecidable in the general case [Moss 2009, "Templates Are Turing Complete"].

**Tentative parsing:** Clang handles the declaration/expression ambiguity via **tentative parsing** — attempting to parse the input as a declaration, using a "tentative" mode that doesn't build AST nodes, and falling back to expression parsing if the tentative parse fails [Chapter 32, `clang/lib/Parse/ParseTentative.cpp`]. This is backtracking recursive descent in a controlled form, applied only at specific grammatical decision points.

### The grammar-action interleaving

A parser generator separates the grammar specification from the semantic actions. In a hand-written parser, the grammar *is* the code — each `parseDeclaration()` function is simultaneously the grammar rule for declarations and the semantic action for building a declaration node.

This interleaving has several advantages:

1. **Arbitrary semantic lookahead:** The parser can call `Sema.lookupType(name)` in the middle of parsing to resolve an ambiguity, then backtrack if the type lookup fails. A grammar-action separation makes this impossible.

2. **Grammar evolution:** Changing a C++ grammar rule in a parser generator requires modifying the grammar file and regenerating tables, then fixing conflicts. Changing a C++ grammar rule in Clang means editing a C++ function. For a language that evolves with every standard release, this is decisive.

3. **Debugging:** A hand-written parser can be stepped through in a debugger. A generated parser's table-driven loop is opaque — debugging requires understanding the generated state machine.

### Error message quality

Recursive-descent parsers can produce more precise error messages because each parsing function knows exactly what it expected. Compare:

```
// Generated LALR parser
error: unexpected token 'else' near line 42

// Clang's hand-written parser
error: expected ';' after return statement
    return x
           ^
           ;
```

Clang's parser knows it is inside a return statement, has seen an expression `x`, and expected `;` after it. The generated parser sees only that the token `else` (or whatever follows the return statement) is not in the current LALR state's action set.

Clang's error recovery messages include fix-it hints — proposed corrections that many editors apply automatically — which require knowing the syntactic context at a level of detail that generated parsers cannot easily provide.

### When parser generators are appropriate

The case against parser generators for production compilers does not extend to all use cases. Parser generators are the right choice for:

- **Simple DSLs:** SQL dialects, configuration languages, command-line option parsers where the grammar is genuinely regular or fits neatly into LALR(1).
- **Rapid prototyping:** A grammar file is faster to write and modify than a recursive-descent parser during early design.
- **Grammar documentation:** A formal grammar file serves as authoritative documentation. Many language specifications describe grammar informally; a working parser generator grammar is more precise.
- **Research compilers:** Studying optimization theory, type systems, or code generation — where the language is simple and the frontend is not the research contribution — benefits from the reduced investment.

yacc and bison remain in use for hundreds of language tools. The GNU make grammar, most POSIX shell parsers, and many SQL parsers use generated LALR parsers. The decision for production compilers of complex languages is clear — but it is not universal.

## 7.11 Chapter Summary

- **Context-free grammars** G = (V, T, P, S) generate exactly the context-free languages, the appropriate formalism for the syntactic level of programming languages. Leftmost and rightmost derivations yield the same parse trees; the tree is the semantic object, the derivation the construction method.

- **Ambiguity** is a grammar property, not a language property. The dangling-else problem is resolved by grammar stratification or by parser-generator shift-prefer rules. Some languages are *inherently ambiguous* (Parikh 1966); no programming language intended for machine parsing should be.

- **LL(1) parsing** requires computing FIRST and FOLLOW sets, constructing a parse table with at most one entry per (nonterminal, terminal) cell, and eliminating left recursion and common prefixes. The stack-based LL(1) algorithm is O(n). LL(*) (Parr 2011) and ALL(*) (Parr et al. 2014) extend LL prediction to handle non-LL(k) grammars via DFA-based or adaptive prediction.

- **LR parsing** builds the parse tree bottom-up via shift and reduce operations on a stack, with the LR(0) automaton (item sets, closure, goto) as the core data structure. SLR(1) uses FOLLOW sets to resolve conflicts; LALR(1) merges LR(1) states with identical cores; full LR(1) handles cases where LALR fails but is rarely needed. LALR(1) is the standard for parser generators (yacc, bison).

- **GLR** extends LR to ambiguous grammars via the Graph-Structured Stack, maintaining all parallel parses simultaneously. O(n³) time; useful for NLP, C++ ambiguity handling, and island parsing.

- **Pratt parsing** handles binary expressions more cleanly than LR theory by encoding operator precedence as integer binding powers and dispatching parse decisions through per-token nud/led functions. O(n) time, no grammar file required, easily extensible to user-defined operators.

- **Earley parsing** handles all CFGs in O(n³) general, O(n²) unambiguous, O(n) for LR(k) grammars (Leo's optimization). The Scanner/Predictor/Completer operations maintain a chart of Earley items indexed by input position.

- **PEG and packrat parsing** replace nondeterministic CFG alternation with ordered choice, guaranteeing unambiguity at the cost of hiding certain language ambiguities. Packrat achieves linear time via O(n × |G|) memoization, trading space for time.

- **Parser combinators** express grammars as composable higher-order functions. The `Parser a = String → [(a, String)]` type in Haskell captures the semantics; monadic composition (Parsec, Megaparsec, attoparsec, nom) provides industrial-strength libraries. Left recursion and error messages remain the principal limitations.

- **Production compilers use hand-written recursive descent** because C++ (and most real languages) is not context-free at the semantic level, because the grammar-action interleaving enables arbitrary semantic disambiguation during parsing, because error messages require detailed syntactic context, and because hand-written parsers are easier to evolve with the language standard. Chapter 32 covers Clang's parser implementation in detail.
